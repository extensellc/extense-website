# DITA / Technical Writing Interview Questions

Job interview questions for DITA and Technical Writing roles, organized by skill level and topic area. Based on the full curriculum of the Ex-tense DITA Training Guide.

---

## Part I — Intelligent Content & Content Strategy

### Beginner

1. What is "intelligent content" and what characteristics distinguish it from traditional documentation?

   > **Answer:** Intelligent content has five defining characteristics: (1) **structurally rich** — organized into typed, modular units (concepts, tasks, references) with semantic markup; (2) **semantically categorized** — metadata describes what it's about, who it's for, difficulty level, and user intent; (3) **discoverable** — metadata, taxonomy, and controlled vocabulary enable precise search/filtering by both humans and AI; (4) **reusable** — authored once and assembled into multiple outputs, audiences, and delivery channels; (5) **adaptable** — conditional processing and metadata-driven filtering enable personalized delivery based on user role, platform, or product version. Traditional documentation is document-centric (the file is the unit), formatting is embedded, metadata is an afterthought, and each document produces one output. Intelligent content is component-centric (the topic is the unit), formatting happens at publish time, metadata is a first-class citizen, and one source produces many outputs.

2. What are the limitations of unstructured formats like Markdown and Word when it comes to machine-readability and AI consumption?

   > **Answer:** AI systems need to (1) **find** content by understanding user intent, not just keyword matching; (2) **filter** by relevance and audience; (3) **extract** focused answers from the right section; (4) **format** appropriately (steps for tasks, prose for concepts, tables for references); and (5) **cite** back to the authoritative source. Unstructured formats fail at all five. They provide no content type information (is it a concept, task, or reference?), no audience metadata (who is this for?), no difficulty level (beginner or advanced?), no intent signal (explain, instruct, or troubleshoot?), no domain classification, fragile chunk boundaries (headings only, no section IDs), no freshness dates for ranking, and no prerequisite chains. The consequences are poor retrieval precision, wrong audience matching, broken intent routing, duplicate/conflicting answers, and no data for measurement or improvement. Compensating requires escalating prompt engineering, increases hallucination risk, and breaks at scale (200+ topics).

3. Describe the five levels of the AI-Readiness Maturity Model (Unstructured → Structured → Enriched → Connected → Intelligent). Where would you place most enterprise documentation today, and why?

   > **Answer:** The five levels are: **Level 1 — Unstructured:** Flat files (Word, wiki, flat XML) with no metadata or typing; AI capability limited to keyword search. **Level 2 — Structured:** Typed units (concept/task/reference) with semantic markup and basic metadata; enables vector search with section-aware chunking and type-based formatting. **Level 3 — Enriched:** Full metadata (audience, difficulty, domain, intent, priority) with controlled vocabulary and section IDs; enables metadata-filtered retrieval, intent-based routing, audience-matched results, and priority-weighted ranking. **Level 4 — Connected:** Content linked in learning paths, prerequisite chains, and relationship tables with glossary synonym resolution and Schema.org output; enables follow-up suggestions, "read this first" guidance, and SEO structured data. **Level 5 — Intelligent:** Analytics-instrumented with feedback loops and self-optimizing delivery based on usage data; enables data-driven gap analysis, automated quality scoring, and continuous improvement. Most enterprise documentation sits at Level 1–2. Many organizations have adopted structured formats but lack the enriched metadata (Level 3) needed for effective AI integration.

4. Why does a component-based content model scale better than a document-centric one?

   > **Answer:** In a document-centric model, the file is the unit — formatted content is embedded, updates require editing each document separately, and each document produces a single output. In a component-based model, the topic is the unit — each topic has a type (concept/task/reference), formatting happens at publish time, and the same topic can be assembled into multiple deliverables. This scales better because: updates to shared content happen once and propagate everywhere (via conref/keyref); the same topics can serve multiple audiences, products, and formats without duplication; conditional processing enables variants without maintaining separate copies; metadata on each component powers filtering, search, and AI consumption; and new deliverables (e.g., adding a chatbot) don't require rewriting content — just adding a new publishing pipeline.

### Intermediate

5. How does DITA's structural typing (concept, task, reference) directly support AI capabilities like intent routing and answer formatting?

   > **Answer:** Each DITA topic type carries a machine-readable signal that AI systems can use in three ways. **Intent routing:** When a user asks "What is X?" the system routes to concept topics; "How do I X?" routes to task topics; "List of X" or "Parameters for X" routes to reference topics. The `topic_type` metadata on each chunk enables this classification. **Answer formatting:** Concept topics produce prose explanations, task topics produce numbered step-by-step instructions extracted from `<steps>`, and reference topics produce structured tables. The AI can format responses appropriately because it knows the content type. **Classification:** Information typing serves as an additional metadata dimension for filtering — a chatbot can exclude reference topics when the user clearly wants conceptual understanding, or prioritize task topics for procedural questions. Without typing, all content is treated identically, and the AI cannot distinguish explanations from procedures from lookup data.

6. A client's documentation exists entirely in Confluence wiki pages. What business case would you make for migrating to structured content?

   > **Answer:** The business case rests on four pillars. **Multi-channel delivery:** Confluence produces one output (wiki pages). Structured DITA produces HTML5 portals, branded PDFs, chatbot knowledge bases, Schema.org web pages, and localized variants — all from a single source. **Reuse and consistency:** Confluence forces copy-paste duplication; when a safety warning changes, you update it in 12 places. DITA's conref/keyref mechanisms mean you update once and it propagates everywhere, reducing maintenance cost and inconsistency risk. **AI readiness:** Confluence provides no structured metadata — no audience, difficulty, intent, or domain classification. DITA's prolog metadata enables filtered retrieval, intent routing, and personalized chatbot answers. The training demonstrates an AI-readiness score improvement from 2.1/10 to 9.2/10 after proper structuring and enrichment. **Scale and governance:** Confluence breaks with large documentation sets (200+ pages become unmanageable). DITA's topic-based architecture, controlled vocabulary via subject schemes, and automated validation in CI/CD pipelines provide governance at scale. Quantifiable KPIs to track include: metadata coverage (0% → 100%), retrievable chunks per query, content reuse instances, variant efficiency, and publication format count.

## Part II — Content Preparation & Tooling

### Beginner

7. What is XML, and how do elements, attributes, and nesting work at a basic level?

   > **Answer:** XML (eXtensible Markup Language) is a text-based format for representing structured data using tags. **Elements** are the building blocks — they have an opening tag (`<title>`), content, and a closing tag (`</title>`). Self-closing tags (`<image href="photo.png"/>`) have no content. **Attributes** are name-value pairs on the opening tag that provide additional information: `<topic id="install" xml:lang="en-US">`. **Nesting** means elements can contain other elements, creating a tree structure: `<ul>` contains `<li>` items, `<steps>` contains `<step>` elements, etc. In DITA, a well-formed XML document has exactly one root element (e.g., `<concept>`, `<task>`, or `<reference>`), every opened tag must be closed, elements must nest properly (no overlapping), attribute values must be quoted, and the document must declare its DOCTYPE to identify which DITA topic type it conforms to.

8. Name the three core DITA topic types and explain when you would use each one.

   > **Answer:** The three core types are: **Concept** (`<concept>`) — for explaining what something is and why it matters. Use for definitions, mental models, background, rationale, architectural overviews, and decision tradeoffs. The body element is `<conbody>`. **Task** (`<task>`) — for teaching how to do something step-by-step. Use for installation procedures, configuration workflows, and checklists. The body element is `<taskbody>`, containing `<prereq>`, `<context>`, `<steps>`, `<result>`, and `<postreq>`. **Reference** (`<reference>`) — for presenting facts, specifications, and lookup data. Use for command syntax, configuration parameters, API specs, error codes, and field definitions. The body element is `<refbody>`, typically containing `<simpletable>` or `<table>` elements. Quick decision guide: *What/Why?* → Concept. *How?* → Task. *Facts to look up?* → Reference.

9. What is the relationship between topics and maps in DITA?

   > **Answer:** **Topics** are self-contained units of information — each topic covers one focused subject and can stand on its own. Topics contain the actual content (explanations, procedures, reference data) but do not define their own sequence, hierarchy, or relationships to other topics. **Maps** (`.ditamap` files) define the publication structure — they organize topics into a table of contents, define hierarchy and nesting, establish key definitions, specify conditional processing rules, and create relationship tables for cross-references. A single topic can be included in multiple maps, enabling reuse across different deliverables (e.g., a "Reset Password" task might appear in both an Admin Guide and a Quick Start Guide). Maps use `<topicref>` elements to reference topics, `<topichead>` for non-linkable groupings, `<topicgroup>` for invisible groupings, and `<keydef>` for key definitions. This separation of content (topics) from context (maps) is fundamental to DITA's reuse and multi-channel publishing model.

10. Describe the steps to install and verify the DITA Open Toolkit. What prerequisites are required?

    > **Answer:** **Prerequisites:** Java 17 or later (verify with `java -version`), 500 MB free disk space, and network access. **Installation steps:** (1) Verify Java — run `java -version`; install JDK 17+ if needed (macOS: `brew install openjdk@17`; Ubuntu: `sudo apt install openjdk-17-jdk`; Windows: download from adoptium.net). (2) Download DITA-OT 4.x from dita-ot.org/download or via curl: `curl -LO https://github.com/dita-ot/dita-ot/releases/download/4.3.5/dita-ot-4.3.5.zip`. (3) Extract to a permanent location: `unzip dita-ot-4.3.5.zip -d /opt/`. (4) Add to PATH: `export DITA_HOME=/opt/dita-ot-4.3.5` and `export PATH="$DITA_HOME/bin:$PATH"` in your shell profile. (5) Verify: `dita --version` should show "DITA-OT version 4.3.5"; `dita --plugins` should list built-in plugins (org.dita.html5, org.dita.pdf2, etc.). (6) Test build: `dita -i $DITA_HOME/docsrc/samples/sequence.ditamap -f html5 -o /tmp/dita-test-output`. Common issues: "command not found" means PATH is wrong; "JAVA_HOME is not set" means you need to export JAVA_HOME; "Unsupported class file major version" means Java is too old.

### Intermediate

11. Compare Oxygen XML Editor and VS Code as DITA authoring environments. What are the tradeoffs?

    > **Answer:** **Oxygen XML Editor** is the industry-standard DITA authoring tool. Strengths: built-in Author mode with visual WYSIWYG editing; real-time DTD/schema validation; DITA Maps Manager for visual map editing; integrated DITA-OT for one-click publishing; refactoring tools for renaming IDs and updating references; Templates and Maps Manager for consistent topic creation; profiling/conditional text visualization; and deep Git integration. Tradeoff: commercial license cost, steeper learning curve for configuration. **VS Code** with XML extensions (e.g., Red Hat XML) is a lighter alternative. Strengths: free and open source; familiar to developers; strong Git integration; extensible via marketplace; good for teams that also write code. Tradeoffs: no native Author mode (you work in raw XML); validation requires extension configuration; no built-in DITA Maps Manager; publishing requires command-line DITA-OT usage; less DITA-specific tooling out of the box. **Recommendation:** Oxygen for dedicated technical writing teams that need visual editing and integrated publishing. VS Code for developer-heavy teams or small projects where cost matters and authors are comfortable with XML source.

12. Explain the difference between `conref` and `keyref`. When would you choose one over the other?

    > **Answer:** **Conref** (content reference) is a direct pull mechanism — it copies an element from a specific file and ID into the current location: `<p conref="reuse/common-notes.dita#common-notes/install-note"/>`. The reference is hard-coded to a specific file path and element ID. It's best for stable, reusable content blocks: legal text, safety warnings, common prerequisites, and shared step fragments. **Keyref** is an indirect, late-binding reference — it resolves through a key defined in the map: `<ph keyref="product-name"/>` resolves to whatever value the map's `<keydef keys="product-name">` specifies. Change the key definition in the map, and the same topic publishes differently without editing. It's best for anything that varies by deliverable: product names, version numbers, URLs, partner branding, and company names. **Choose conref** when the reused content is the same everywhere (e.g., a safety warning that never varies). **Choose keyref** when the value varies by context or deliverable (e.g., product name differs between editions). **Conkeyref** combines both: it pulls content via a key rather than a direct file path, enabling the same topic to pull different snippets depending on which map it's built from.

## Part III — Authoring DITA Content

### Beginner

13. What are the main structural elements of a DITA task topic? Walk through a well-formed `<task>` from `<prereq>` to `<postreq>`.

    > **Answer:** A DITA task topic has the following structure: `<task id="install-dita-ot">` → `<title>` (what the procedure achieves) → `<shortdesc>` (one-sentence summary) → `<taskbody>` containing: (1) `<prereq>` — conditions, permissions, or dependencies required before starting; (2) `<context>` — when and why you perform this task; (3) `<steps>` — the ordered procedure, with each `<step>` containing a `<cmd>` (imperative action), optional `<info>` (supplementary detail), and optional `<stepresult>` (what happens after the step); (4) `<result>` — what success looks like after completing all steps; (5) `<postreq>` — what to do next or related follow-up tasks. Best practices: use imperative verbs in `<cmd>` ("Click", "Run", "Verify"), limit each step to one testable action, separate UI and CLI procedures if they diverge significantly, and use semantic elements like `<uicontrol>`, `<menucascade>`, `<filepath>`, and `<codeph>` for UI and code references.

14. When writing topic titles, why should you avoid titles containing "and" or "then"?

    > **Answer:** Titles with "and" or "then" signal that the topic covers multiple goals and should likely be split. For example, "Install and Configure the System" combines two distinct tasks — installation and configuration — that have different prerequisites, steps, and results. Splitting into two topics ("Installing the System" and "Configuring the System") makes each topic more focused, more reusable (a user may need one but not the other), and more retrievable (search engines and AI systems can match user intent to a single focused topic). Similarly, "What It Is and How to Use It" mixes a concept with a task — splitting gives proper information typing. This heuristic is part of the chunking decision: each topic should solve one user goal and be understandable in isolation.

15. What is "chunking" in the context of topic authoring, and what factors influence how granular your topics should be?

    > **Answer:** Chunking is the decision about how large or small to make individual topics. It's the most important authoring decision because it affects reuse, translation cost, search relevance, link stability, and maintenance. **Make a topic** when content solves one user goal, can be reused elsewhere, or could appear in search results on its own. **Make a section** (not a separate topic) when content is only meaningful in the same context and splitting would create "orphan topics" with unclear purpose. **Split a topic** if it covers multiple distinct goals, is too long to scan, mixes concept and procedure, or gets reused partially. Guidelines for AI-ready content: topics should be 500–3,000 words with 3–8 sections; each section should address one question at 100–1,200 tokens; `<shortdesc>` should be one sentence at 25–35 words; and section titles should be phrased as questions when possible ("How does conref work?" matches user queries better than "Conref Overview").

### Intermediate

16. You have a document that explains what a feature is, gives step-by-step setup instructions, and lists all configuration parameters in a table. How would you split this into DITA topics, and why?

    > **Answer:** Split into three topics matching DITA's information types: (1) A **concept topic** — "Understanding [Feature]" — that explains what the feature is, why it matters, and when to use it. Body uses `<conbody>` with `<section>` elements for key ideas. (2) A **task topic** — "Setting Up [Feature]" — with `<prereq>`, `<steps>`, and `<result>` covering the step-by-step procedure. (3) A **reference topic** — "[Feature] Configuration Parameters" — with a `<simpletable>` listing each parameter, its description, allowed values, and defaults. **Why:** Each topic type has a distinct purpose that enables proper validation (tools enforce the correct content model), appropriate formatting (PDF and HTML render each type differently), AI capability (chatbots can route "What is X?" to the concept, "How do I X?" to the task, and "What options does X have?" to the reference), reuse (the reference can be reused in an API guide without the setup instructions), and maintenance (updating parameters doesn't require touching the procedure).

17. Explain the role of stable IDs in DITA. What breaks when IDs change?

    > **Answer:** IDs in DITA serve as stable addresses for content — they should be treated like a public API. **File names** should use lowercase with hyphens (`install-windows.dita`) and remain stable over time. **Topic IDs** should match the filename or a semantic label (`id="install"` in `install.dita`). **Element IDs** on reuse targets should be meaningful (`id="install-note"`, `id="admin-warning"`). When IDs change, several things break: **conref targets** — any topic referencing `<p conref="file.dita#topic/old-id"/>` gets a build error; **keyref targets** — key definitions pointing to the old ID fail; **cross-references** — `<xref>` links to the renamed topic or element produce broken links or warnings (DOTX036W, DOTJ031E); **external links** — bookmarks, URLs, and citations from outside the DITA system become dead links; **translation memory** — TMS segments keyed on topic ID lose their matches, increasing retranslation cost. The stability rule: change IDs rarely, communicate changes, update all dependent references, and validate builds after renaming.

18. How does `conkeyref` combine the benefits of both `conref` and `keyref`? Give a practical example.

    > **Answer:** `conkeyref` pulls content (like conref) but resolves the source file through a key (like keyref), enabling the same topic to pull different reusable content depending on which map it's built from. **Practical example:** Suppose you have a shared installation task that includes a legal notice, but the legal text differs per product edition. (1) Define a key in each map: in `product-a.ditamap`, `<keydef keys="legal-install-note" href="reuse/product-a-legal.dita"/>`; in `product-b.ditamap`, `<keydef keys="legal-install-note" href="reuse/product-b-legal.dita"/>`. (2) Each snippet file has a matching structure with the same element ID (`id="install-legal"`). (3) In the shared task topic: `<note conkeyref="legal-install-note/install-legal"/>`. Now building with `product-a.ditamap` pulls Product A's legal text, and building with `product-b.ditamap` pulls Product B's — no editing of the shared topic required. This is powerful for multi-product or partner-branded documentation.

19. Describe how DITAVAL filtering works. How would you produce audience-specific outputs (e.g., for developers vs. technical writers) from a single source?

    > **Answer:** DITAVAL filtering works in three steps: (1) **Tag content** with profiling attributes in your topics: `<section audience="developer">`, `<step platform="windows">`, `<note product="pro">`. (2) **Create DITAVAL files** that define include/exclude/flag rules for each attribute value. (3) **Apply the filter at build time** with `--filter=path/to/filter.ditaval`. For audience-specific outputs, create two DITAVAL files: **developer.ditaval:** `<val><prop action="include" att="audience" val="developer"/><prop action="include" att="audience" val="all"/><prop action="exclude" att="audience" val="technical-writer"/></val>`. **writer.ditaval:** `<val><prop action="include" att="audience" val="technical-writer"/><prop action="include" att="audience" val="all"/><prop action="exclude" att="audience" val="developer"/></val>`. Build twice: `dita -i guide.ditamap -f html5 -o out/dev --filter=developer.ditaval` and `dita -i guide.ditamap -f html5 -o out/writer --filter=writer.ditaval`. You can also use `action="flag"` to visually mark content (e.g., with an icon) rather than excluding it. Best practice: tag at the coarsest level possible (whole sections or steps, not individual sentences) and use controlled vocabulary via a subject scheme to prevent inconsistent attribute values.

20. You're building documentation for a product that ships on three platforms (Windows, macOS, Linux). Outline a reuse strategy using keys and conditional processing.

    > **Answer:** **Strategy overview:** Combine keys for context-varying references and DITAVAL filtering for platform-specific content. **Keys:** Define platform-specific values in each map variant (or use key scopes): `<keydef keys="install-path"><topicmeta><keywords><keyword>/opt/myapp</keyword></keywords></topicmeta></keydef>` for Linux vs. `C:\Program Files\MyApp` for Windows. Use `<ph keyref="install-path"/>` in topics instead of hard-coding paths. Similarly, key download URLs, package manager commands, and service names. **Conditional processing:** Tag platform-specific steps with `platform="windows"`, `platform="macos"`, or `platform="linux"`. Keep conditional blocks coarse — tag entire steps or sections, not individual words. If more than half a topic differs by platform, split into separate platform-specific task topics and assemble differently per map. **DITAVAL files:** Create `windows.ditaval`, `macos.ditaval`, `linux.ditaval` — each includes its platform and excludes the others. **Build:** Either three separate builds with different filters, or use a DITA project file with three deliverables sharing the same context but with different filter profiles. **Governance:** Define allowed platform values in a subject scheme to prevent authors from inventing `win`, `Win`, `windows`, `win32` as separate values.

### Advanced

21. How does DITA information typing (concept/task/reference) influence AI chatbot behavior — specifically intent routing, answer formatting, and classification?

    > **Answer:** **Intent routing:** The chatbot classifies user questions by intent and routes them to matching topic types. "What is X?" → concept topics (intent: explain). "How do I X?" → task topics (intent: how-to). "X is not working" → troubleshooting sections. "List of X" / "Parameters for X" → reference topics. The `topic_type` and `content-intent` metadata on each chunk enable this routing. Without typing, all content is treated identically and the system cannot distinguish explanations from procedures. **Answer formatting:** The chatbot formats responses based on content type — concept chunks produce prose explanations, task chunks produce numbered steps (extracted from the `<steps>` structure), and reference chunks produce formatted tables or lists. This dramatically improves answer quality compared to generic text blocks. **Classification:** Information typing adds a dimension to vector search filtering. When a user asks a procedural question, the system can boost or exclusively search task-type chunks, eliminating conceptual content that would dilute the answer. Combined with audience and difficulty metadata, this creates multi-faceted retrieval: e.g., "Show me beginner tasks for technical writers about publishing" filters on `difficulty=beginner AND audience=technical-writer AND content-domain=publishing AND topic_type=task`. Testing showed cross-encoder reranking combined with metadata filtering improved answer accuracy 18–24% vs. vector search alone.

## Part IV — Information Architecture

### Intermediate

22. What are the key considerations when designing a DITA map hierarchy? How does map-driven architecture differ from a file-based directory approach?

    > **Answer:** Key considerations: (1) **Topics are self-contained** — every topic must be understandable without surrounding context because a chatbot may retrieve a single section from a single topic. (2) **Maps define context, not topics** — sequence, hierarchy, and relationships are defined in the map, not hard-coded into topics, allowing the same topic to appear in multiple learning paths. (3) **Progressive depth** — content organized from foundational to advanced with explicit prerequisite chains. (4) **Granularity for retrieval** — each topic covers one focused subject; broad topics produce low-quality chunks. (5) **Role-based navigation** — audience metadata + conditional processing enable filtered views for different roles. **Map-driven** architecture separates organization from content: the map hierarchy signals sequence (read in this order), nesting (sub-topics elaborate on parent), and grouping (related content clusters). The same topics can be reorganized into different deliverables by creating different maps. A **file-based directory** approach embeds organization in the folder structure, coupling content to one hierarchy and making it difficult to create alternate views or deliverables without duplicating files.

23. What is a DITA subject scheme, and how does it enforce controlled vocabulary across a content set?

    > **Answer:** A subject scheme is a specialized DITA map (`<subjectScheme>`) that formally defines the allowed values for classification attributes. It uses `<subjectdef>` elements to declare valid values organized hierarchically, and `<enumerationdef>` to bind those values to specific attributes. For example, a subject scheme can declare that the `audience` attribute only accepts values "technical-writer", "developer", "build-engineer", "manager", and "all". **Enforcement:** When bound to a map, DITA-aware editors like Oxygen XML validate attribute values against the scheme — if an author types `audience="dev"` instead of `audience="developer"`, the editor flags it as invalid. This prevents the inconsistency problem where authors invent values freely (`developers`, `dev`, `engineers` treated as three different audiences). The scheme is machine-readable, version-controlled, and shareable across projects, serving as a governance tool and a chatbot's metadata validation reference.

24. Name the five classification facets used in this training's taxonomy (audience, difficulty, content domain, content intent, service line). Why does AI need controlled vocabulary rather than free-text tags?

    > **Answer:** The five facets are: (1) **Audience** (technical-writer, developer, build-engineer, manager, all) — filter results by user role, personalize recommendations. (2) **Difficulty** (beginner, intermediate, advanced, expert) — match content level to user expertise, suggest simpler alternatives. (3) **Content domain** (dita-fundamentals, authoring, publishing, customization, xslt, devops) — domain-scoped retrieval; only search relevant subject areas. (4) **Content intent** (explain, how-to, reference, troubleshoot, best-practice, tutorial) — intent routing; match user question type to content type. (5) **Service line** (dita-engineering, publishing-engineering, xml-engineering, training) — link content to business service offerings. **Why controlled vocabulary matters for AI:** Free-text tags produce inconsistency — "conditional processing", "DITAVAL filtering", "content filtering", and "profiling" might all mean the same thing but appear as four different values. When a chatbot filters by metadata, inconsistent values mean relevant content is missed. Controlled vocabulary ensures every chunk uses the exact same normalized terms, making metadata filters reliable. The subject scheme serves as the single source of truth for allowed values.

### Advanced

25. How do you bind a subject scheme to a map so that editors can validate metadata values? What happens when an author uses a value outside the scheme?

    > **Answer:** You bind a subject scheme to a map by referencing it from the map's `<subjectScheme>` processing instruction or by including it as a submap reference. In Oxygen XML Editor, you can associate the scheme through the DITA Maps Manager or via a `<mapref>` to the scheme file. The scheme uses `<enumerationdef>` to bind subject definitions to specific attributes — e.g., binding the "audience-type" subject hierarchy to the `@audience` attribute. **When an author uses an invalid value:** DITA-aware editors display a validation error or warning, similar to using an invalid attribute value against a DTD. The author sees that the value is not in the allowed list. In Oxygen, the editor can provide a dropdown of valid values from the scheme, preventing invalid entries entirely. At build time, DITA-OT may issue warnings about unrecognized attribute values if strict processing mode is enabled. This creates a governance layer: the taxonomy is enforced at authoring time, not just at build time, catching errors early.

26. How do taxonomy and metadata normalization impact downstream AI systems like RAG pipelines and chatbots?

    > **Answer:** The impact is direct and measurable across the RAG pipeline: **At ingestion:** Each chunk inherits normalized metadata from its topic's prolog. Controlled vocabulary ensures "audience=developer" is identical across all 210+ chunks, not mixed with "dev", "developers", "engineers". **At retrieval:** Metadata-filtered vector search relies on exact-match filters. A query filtered to `content-domain=publishing AND difficulty=beginner` only works if every relevant chunk uses those exact values. Without normalization, relevant chunks with `domain=pub` or `level=101` are invisible. The training shows metadata filters eliminate 95%+ irrelevant chunks. **At reranking:** The `chatbot-priority` field (high/medium/low) applies boost multipliers (1.3×, 1.0×, 0.8×) to retrieval scores. Inconsistent priority values would corrupt ranking. **At generation:** Intent routing maps user question patterns to `content-intent` values — "What is X?" → `explain`, "How do I X?" → `how-to`. Without normalized intents, the router cannot function. **Net impact:** The Extense case study showed metadata enrichment with controlled vocabulary raised the AI-readiness score from 7.2/10 to 9.2/10 — the difference between partial capability and production-ready chatbot deployment.

## Part V — Metadata Enrichment & Migration

### Intermediate

27. Walk through the nine-step migration process for converting unstructured content into properly typed DITA topics.

    > **Answer:** The nine steps are: **(1) Audit source content and classify by information type** — read every source file and determine its primary purpose: explains a concept → `<concept>`, teaches a procedure → `<task>`, provides lookup data → `<reference>`. Create a spreadsheet mapping source files to DITA types. **(2) Convert structural markup** — transform HTML/flat XML elements to DITA equivalents: `<h1>` → `<title>`, `<h2>` → `<section><title>`, `<ol><li>` → `<steps><step>`, `<pre>` → `<codeblock>`, `<code>` → `<codeph>`, `<a>` → `<xref>`. **(3) Write `<shortdesc>` for every topic** — one clear sentence (25–35 words) that describes the topic's purpose. Critical for AI — prepended to every chunk as summary context. **(4) Add `xml:lang="en-US"` to every root element** — enables NLP language detection in the RAG pipeline. **(5) Add full AI-ready prolog metadata** — the most impactful step: `<author>`, `<critdates>`, `<keywords>` with `<indexterm>`, `<audience>`, `<category>`, `<resourceid>`, and custom `<data>` elements for difficulty, duration, content-domain, content-intent, chatbot-priority, service-line, and prerequisites. **(6) Ensure every `<section>` has `@id` and `<title>`** — sections without IDs can't be deep-linked; sections without titles produce unlabeled chunks. **(7) Add `outputclass="language-*"` to all codeblocks** — enables syntax highlighting and language detection in RAG chunks. **(8) Build DITA map** defining topic organization and learning paths. **(9) Validate** by building HTML5 and testing chatbot ingestion.

28. Describe a complete AI-ready `<prolog>` block. What elements and custom `<data>` fields should it contain?

    > **Answer:** A complete AI-ready prolog contains: `<author>` — content owner (e.g., "Extense LLC"); `<critdates>` with `<created date="2026-02-10"/>` and optional `<revised>` — content freshness for ranking; `<metadata>` containing `<keywords>` with 3–6 `<indexterm>` elements for search matching, `<category>` for taxonomy classification, and `<audience>` for target role; `<resourceid appid="extense-training" appname="chatbot" id="topic-id"/>` — unique identifier for the chatbot system. Custom `<data>` elements: `<data name="difficulty" value="intermediate"/>` — beginner/intermediate/advanced/expert for level-appropriate retrieval; `<data name="duration" value="PT20M"/>` — ISO 8601 reading/completion time; `<data name="content-domain" value="publishing"/>` — subject area for domain-filtered retrieval; `<data name="content-intent" value="explain"/>` — explain/how-to/reference/troubleshoot/tutorial for intent routing; `<data name="chatbot-priority" value="high"/>` — high/medium/low for retrieval ranking weight; `<data name="service-line" value="publishing-engineering"/>` — linked business service; `<data name="prerequisites" value="previous-topic-id"/>` — learning path sequencing. All values must come from the controlled vocabulary defined in the subject scheme.

29. What is the difference between manual and scripted metadata enrichment? When would you choose each approach?

    > **Answer:** **Manual enrichment:** Open each file in an editor, read the content, determine appropriate metadata values for each field, and add the full `<prolog>` block by hand. Advantages: contextually accurate — a human reads each topic and makes nuanced decisions about audience, difficulty, intent, and priority. Best for: small content sets (<10 topics), initial taxonomy design when you're establishing patterns, and topics requiring careful judgment about classification. **Scripted enrichment:** A Python script reads each file, determines topic type from the root element, and replaces or injects a full prolog block with metadata values from a spreadsheet or configuration. The Extense training uses `enrich-metadata.py` for batch processing. Advantages: fast, consistent, and repeatable — can process hundreds of topics in a single operation. Best for: large migrations (50+ topics), when metadata patterns are well-established, and for ensuring uniform structure across a corpus. **In practice:** Use manual enrichment to establish patterns on the first 5–10 topics, document those patterns, then script the remaining batch. Verify scripted results with `grep` commands: `grep -rl 'chatbot-priority' dita-topics/ | wc -l` to count enriched files.

30. What does "100% metadata coverage" mean, and why is it critical for AI-powered content delivery?

    > **Answer:** 100% metadata coverage means every single topic in the content set carries all required metadata fields — audience, difficulty, content-domain, content-intent, chatbot-priority, service-line, critdates, keywords, resourceid, and xml:lang. **Why it's critical:** In a RAG pipeline, metadata-filtered retrieval narrows the search space before vector similarity is computed. If a topic is missing `audience` metadata, it becomes invisible to any query filtered by audience — the chatbot simply won't retrieve it for audience-specific questions. The Extense case study demonstrated this: before enrichment, 69% of topics (34 of 49) were invisible to metadata-filtered retrieval because they lacked the required fields. After achieving 100% coverage, all 210+ chunks became addressable by every metadata dimension. The AI-readiness score jumped from 7.2/10 to 9.2/10. Partial coverage creates a worse outcome than no coverage — users get inconsistent results depending on whether their query happens to match enriched or unenriched topics.

### Advanced

31. You've been given 500 Markdown files to migrate to DITA. Outline your strategy for topic typing, ID assignment, metadata enrichment, and validation.

    > **Answer:** **Phase 1 — Audit and classify (1–2 weeks):** Scan all 500 files and classify each by information type. Use scripting to detect patterns: files with numbered steps → likely tasks; files with tables of parameters → likely references; otherwise → concepts. Create a mapping spreadsheet with filename, target DITA type, proposed topic ID, and target metadata values. Identify files that mix types and need splitting. **Phase 2 — Topic typing and ID assignment:** Assign stable, meaningful IDs matching filenames (lowercase with hyphens: `install-windows`). 32. How do DITA topic types map to Schema.org structured data? What are the SEO and AI-agent implications?

    > **Answer:** The mappings are: `<concept>` → **TechArticle** (with headline, description, author, datePublished, proficiencyLevel, dependencies); `<task>` → **HowTo** (with name, description, step[].text, totalTime, tool — steps extracted from `<steps>`); `<reference>` → **TechArticle** (with articleSection additions for API docs); `<glossentry>` → **DefinedTerm** (with name, description, termCode, inDefinedTermSet); `<glossgroup>` → **DefinedTermSet**; `<bookmap>` → **Course** (with provider, hasCourseInstance, coursePrerequisites). Glossary entries generate DefinedTerm cards. Concept topics include expertise level indicators. **AI-agent implications:** Schema.org structured data makes content discoverable by AI agents and search engines beyond your own chatbot. The `about` and `audience` properties enable external AI systems to evaluate content relevance without full-text analysis.

33. The training reports an AI-readiness score improvement from 2.1 to 9.2 after metadata enrichment. What specific metadata fields drive that improvement, and how would you measure it on your own project?

    > **Answer:** The improvement is driven by gains across the ten AI-readiness dimensions: **(1) Metadata completeness** (6/10 → 9/10) — adding all 8 required fields (audience, difficulty, content-domain, content-intent, chatbot-priority, service-line, critdates, keywords) to every topic. **(2) Consistency** (5/10 → 9/10) — standardizing values via subject scheme (previously 69% of topics invisible to filtered retrieval). **(3) Taxonomy and controlled vocabulary** — formalizing the subject scheme with 400+ taxonomy-controlled values. **(4) Retrieval optimization** — clear shortdescs, section titles as questions, front-loaded information. **(5) AI/RAG pipeline readiness** — section IDs on every section, configuring chunking parameters. The largest single driver is metadata completeness — going from 0% to 100% coverage makes all chunks addressable. **To measure on your own project:** Score each of the 10 dimensions (structural intelligence, metadata completeness, RAG readiness, taxonomy, glossary, Schema.org, retrieval optimization, reuse architecture, consistency, scalability) on a 1–10 scale. Automated checks: count files with all required `<data>` elements; verify attribute values against subject scheme; measure percentage of sections with IDs; track conref/keyref usage rates; run ingestion pipeline and inspect the report for missing metadata warnings. Target: 8+/10 on all dimensions for production chatbot deployment.

## Part VI — Publishing & Automation

### Beginner

34. What is the DITA Open Toolkit, and what role does it play in the publishing pipeline?

    > **Answer:** The DITA Open Toolkit (DITA-OT) is the open-source publishing engine for DITA content. It takes a DITA map (with its referenced topics) as input and produces formatted output in various formats (HTML5, PDF, EPUB, etc.). Its role in the pipeline is to process the five stages of publishing: (1) **preprocess** — parse the map, resolve topicrefs, collect key definitions; (2) **reference resolution** — apply conref pulls, resolve keyrefs, validate IDs; (3) **filtering** — include/exclude content based on DITAVAL rules; (4) **transform** — generate the target format (HTML, XSL-FO for PDF, etc.) using XSLT stylesheets; (5) **postprocess** — create navigation assets, copy images, apply styling. DITA-OT never modifies source files — all work happens in a temp directory. It's extensible through plugins and configurable through parameters, properties files, and project files. The current version is 4.x, requiring Java 17+.

35. Name at least four output formats that DITA-OT can produce from a single DITA map.

    > **Answer:** Built-in transtypes: (1) **html5** — modern HTML5 with responsive navigation, search, and CSS support, used for documentation portals; (2) **pdf** — XSL-FO pipeline via Apache FOP, for print-ready and regulatory documents; (3) **xhtml** — legacy XHTML 1.0 output for older CMS platforms; (4) **markdown** / **markdown_github** — reverse transform back to Markdown for GitHub READMEs. Plugin-based formats: (5) **epub** (via org.dita.epub) — EPUB 3 eBooks; (6) **normalize** (via org.dita.normalize) — fully resolved DITA for CMS import. Custom formats can also be created via plugin development, such as branded HTML5 (`com.extense.html5.branded`) or branded PDF (`com.extense.pdf.branded`). All these outputs are produced from the same DITA map and source topics — the single-source, multi-channel publishing model.

### Intermediate

36. Explain the stages of a DITA-OT build pipeline: preprocess → reference resolution → filtering → transform → postprocess.

    > **Answer:** **(1) Preprocess:** Parse the DITA map, resolve `<topicref>` elements, collect key definitions from `<keydef>` elements, and generate file lists and job metadata. This stage also handles chunking decisions (`@chunk` attributes on topicrefs). **(2) Reference resolution:** Apply conref pulls (copy content from referenced elements), conref pushes, resolve keyrefs to their defined values, resolve coderefs (external code file inclusions), and validate that all referenced IDs exist. **(3) Filtering:** Apply DITAVAL rules to include, exclude, or flag content based on profiling attributes (audience, platform, product, etc.). Flagged content may receive visual markers (icons, change bars). **(4) Transform:** Run format-specific XSLT stylesheets to generate the target output. For HTML5, this transforms DITA elements to HTML elements with CSS classes. For PDF, it generates XSL-FO intermediate format, then processes through Apache FOP to produce the final PDF. **(5) Postprocess:** Copy static assets (CSS, images, JavaScript), generate navigation artifacts (TOC pages, sidebar nav), create index pages, and organize the output directory. The temp directory contains intermediate artifacts from each stage and is the primary debugging resource when builds produce unexpected results.

37. What is a DITA project file, and how does it differ from a shell script or properties file for configuring builds?

    > **Answer:** A DITA project file is a declarative XML configuration (using the `https://www.dita-ot.org/project` namespace) that defines one or more build deliverables. Its core elements are: `<context>` (names the input map and shared settings), `<deliverable>` (defines one output, including transtype and output directory), and `<publication>` (specifies transtype and parameter overrides via `<param>` elements). **vs. properties files:** Properties files are simple key-value pairs (e.g., `nav-toc=full`) but can only define one deliverable, have no shared context concept, and cannot compose or include other files. **vs. shell scripts:** Scripts are imperative and platform-specific, requiring maintenance of command-line arguments. Project files are declarative, cross-platform, and self-documenting. **Advantages of project files:** Multiple deliverables in one file (e.g., HTML5 + PDF from the same map); shared contexts so multiple deliverables reuse the same input map; composability via `<include href="other-project.xml"/>`; and native DITAVAL profile support via `<profile>` elements rather than `--args.filter` flags. Run with: `dita --project=build-project.xml`.

38. How would you customize HTML5 output branding (colors, fonts, logos) without writing a plugin?

    > **Answer:** Three techniques, none requiring a plugin: **(1) CSS files:** Create a custom CSS file using CSS variables for brand colors and styling. Set `:root { --brand-primary: #1a3a5c; --brand-accent: #e8721c; }` and style elements like `.topictitle1`, `pre.codeblock`, `nav[role="toc"]`. **(2) Properties file:** Configure build parameters to inject CSS: `args.css=brand.css`, `args.cssroot=css/`, `args.copycss=yes`. Add headers/footers: `args.hdr=includes/header.xml`, `args.ftr=includes/footer.xml`. Control navigation: `nav-toc=full`. Enable features: `args.gen.task.lbl=YES`, `args.indexshow=yes`. **(3) DITAVAL flagging:** Use `action="flag"` to add visual markers to specific content, such as deprecated status icons: `<prop action="flag" att="status" val="deprecated"><startflag imageref="icons/warning.png"><alt-text>Deprecated</alt-text></startflag></prop>`. This approach handles 80%+ of branding needs. You only need a plugin when you need to change the HTML structure itself (e.g., adding `<meta>` tags, modifying navigation markup, or wrapping elements in custom `<div>` containers).

39. Describe a CI/CD publishing pipeline for DITA content using GitHub Actions. What triggers the build, and what artifacts does it produce?

    > **Answer:** **Triggers:** Push to `main` branch (for production publishing) and pull requests against `main` (for validation), scoped to content paths: `paths: ['docs/**', 'maps/**', 'topics/**']`. **Pipeline stages:** (1) **Checkout** the repository. (2) **Build environment** — use the official DITA-OT Docker container: `container: { image: ghcr.io/dita-ot/dita-ot:4.3 }`. (3) **Install custom plugins:** `dita --install plugins/com.extense.html5.branded`. (4) **Build all deliverables:** `dita --project=build-project.xml -v` or individual format commands. (5) **Upload artifacts** — use `actions/upload-artifact@v4` to store HTML5 and PDF outputs. (6) **Deploy** — for main branch pushes, deploy HTML5 to GitHub Pages using `peaceiris/actions-gh-pages@v3`. **Validation gates** in the pipeline: XML well-formedness (`xmllint --noout`), link checking (DITA-OT reports broken topicrefs and conrefs), metadata completeness (custom scripts checking required `<data>` elements), and optionally strict processing mode (`--processing-mode=strict`).

### Advanced

40. A DITA build is producing broken cross-references in the output. Walk through your diagnostic process, including temp directory inspection and log analysis.

    > **Answer:** **Step 1 — Enable verbose/debug logging:** Rebuild with `-v` (verbose) or `-d` (debug) and pipe to a log file: `dita -i map.ditamap -f html5 -o out/ -v 2>&1 | tee build.log`. **Step 2 — Search for error codes:** Look for `DOTX036W` (target topic not available — fix the href path), `DOTJ013W` (file not found — check relative path), `DOTJ031E` (conref target not found — verify file path and element ID), `DOTA069W` (key not resolved — add keydef to root map). **Step 3 — Inspect the temp directory:** Use `--temp=/tmp/dita-temp` to specify a temp location. Examine the preprocessed versions of topics — conrefs and keyrefs should be resolved. If a conref target shows as unresolved, the source file path or element ID is wrong. Check the `.job.xml` file for the complete file list and processing metadata. **Step 4 — Verify source references:** Open the referring topic and confirm the `<xref href="...">` path is correct relative to the map or topic. Check for case sensitivity issues (common on Linux), file extensions, and fragment identifiers (`#topic-id/element-id`). **Step 5 — Check for filtering issues:** If references point to elements that are excluded by DITAVAL filtering, they'll break. Build without the filter to confirm, then adjust either the filter rules or the reference locations. **Step 6 — Check for chunking side effects:** The `@chunk` attribute can split or merge topics, changing the output file structure and breaking expected references.

41. How would you set up multi-variant publishing (e.g., different products, audiences, platforms) in a single CI/CD pipeline?

    > **Answer:** Use a DITA project file with multiple deliverables sharing a common context but with different filter profiles. **Project file structure:** Define a shared `<context>` referencing the root map. Create separate `<deliverable>` elements for each variant, each with its own output directory and DITAVAL filter profile: e.g., "Developer HTML5" with `developer.ditaval`, "Writer HTML5" with `writer.ditaval`, "Developer PDF", etc. **DITAVAL files:** One per variant, each including/excluding based on profiling attributes. **CI/CD pipeline:** A single `dita --project=build-project.xml` command builds all deliverables. The GitHub Actions workflow uploads each variant's output as a separate artifact and deploys to different subdirectories or subdomains. **Docker image:** Pre-bake custom plugins into a custom Docker image so plugin installation doesn't slow down every build: `FROM ghcr.io/dita-ot/dita-ot:4.3` → `COPY plugins/ /opt/app/plugins/` → `RUN dita --install`. **Scaling:** For very large builds (2,000+ topics), parallelize by running format-specific builds in separate CI jobs that share the same checkout, or use build caching to avoid redundant preprocessing.

## Part VII — DITA-OT Plugin Development

### Intermediate

42. What is a DITA-OT plugin, and why is creating a plugin preferred over modifying DITA-OT source files directly?

    > **Answer:** A DITA-OT plugin is a self-contained extension package that hooks into the toolkit's build pipeline through declared extension points. It typically contains XSLT overrides, Ant targets, CSS, JavaScript, and/or other resources, organized in a directory with a `plugin.xml` descriptor. **Why plugins are preferred:** (1) **Upgrade safety** — when you upgrade DITA-OT, your core files get replaced. Plugins survive upgrades because they live in the `plugins/` directory and are integrated via `dita --install`. Directly modifying toolkit files means losing customizations on every upgrade. (2) **Minimal overrides** — plugins import the default stylesheets and override only specific templates, keeping the remaining default behavior intact. Copy-paste modification duplicates hundreds of lines that must be manually maintained. (3) **Portability** — plugins can be shared across projects, teams, and CI/CD environments. They're installed with `dita --install path/to/plugin`. (4) **Composability** — multiple plugins can coexist, each customizing different aspects (one for HTML5 styling, another for TOC navigation, another for metadata injection).

43. Describe the structure of a `plugin.xml` file. What are the key elements it must contain?

    > **Answer:** A `plugin.xml` must contain: (1) `<plugin id="com.yourco.html5.custom" version="1.0.0">` — the root element with a unique dot-delimited identifier (reverse-domain convention) and version number. (2) `<require plugin="org.dita.html5"/>` — declares dependencies to ensure correct plugin load order. (3) `<feature>` elements declaring which extension points the plugin uses: e.g., `<feature extension="dita.xsl.html5" file="xsl/html5/override.xsl"/>` to override the main HTML5 XSLT. A typical plugin directory structure: `com.yourco.html5.custom/` containing `plugin.xml`, `xsl/html5/override.xsl`, optionally `xsl/toc/toc-override.xsl`, `css/brand.css`, and `js/site.js`. Install with: `dita --install path/to/com.yourco.html5.custom`.

44. What are DITA-OT extension points? Name three and explain when you would use each.

    > **Answer:** Extension points are named hooks in the DITA-OT pipeline where plugins can inject custom behavior. They're declared in `plugin.xml` via `<feature extension="...">`. Three key extension points: (1) **`dita.xsl.html5`** — override the main HTML5 XSLT stylesheet. Use when you need to change how DITA elements are rendered as HTML (e.g., custom note callout styling, adding `data-` attributes, wrapping elements in custom `<div>`s). (2) **`dita.xsl.html5.toc`** — override the TOC generation XSLT. Use when you need to customize sidebar navigation structure, add CSS classes to TOC elements, or implement collapsible menus. (3) **`depend.preprocess.post`** — run a custom Ant target after preprocessing completes but before the format-specific transform. Use when you need to modify the preprocessed content (e.g., inject metadata, validate content, or transform intermediate files). Other notable points: `depend.preprocess.pre` (before preprocessing), `dita.conductor.html5.param` (pass parameters to the HTML5 transform), `dita.conductor.html5.toc.param` (pass parameters to TOC transform).

45. Explain the difference between `preprocess` and `preprocess2` in DITA-OT 4.x. What is map-first preprocessing?

    > **Answer:** **`preprocess`** (legacy) processes maps and topics in an interleaved fashion — it works through the resolution steps for both maps and topics together. **`preprocess2`** (map-first) completes most map processing first before processing topics. This means key resolution, scope handling, and map-level metadata collection happen upfront, then topic-level processing (conref resolution, filtering, chunking) happens with complete map context available. **Benefits of map-first:** Simplifies key/scope handling because all keys are resolved before any topic processing begins; enables topic-level filtering optimization; is more memory-efficient than legacy preprocess. **DITA-OT 4.x adoption:** PDF and HTML Help have used map-first since 3.0; HTML5 uses map-first as of 4.2. **Developer impact:** Some extension points available in legacy `preprocess` (e.g., granular step hooks like `depend.preprocess.conref.pre`) may not apply in `preprocess2`. Plugin developers should use the high-level hooks (`depend.preprocess.pre`, `depend.preprocess.post`) for stability across both pipelines.

### Advanced

46. Walk through the 11-step DITA-OT preprocessing pipeline. Why are these implemented as separate Ant targets rather than a single monolithic process?

    > **Answer:** The 11 preprocessing steps (as Ant targets): (1) **`clean-temp`** — initialize the working directory. (2) **`copy-files` / `copy-image` / `copy-subsidiary`** — stage source resources into the temp directory. (3) **`gen-list`** — create file lists and job metadata (`.job.xml`). (4) **`mapref`** — resolve map-to-map references (submaps). (5) **`mappull` / `topicpull`** — collect metadata from maps and topics (navtitles, shortdescs). (6) **`keyref`** — resolve all key references to their defined values. (7) **`conref` / `conrefpush`** — resolve content references (pull and push patterns). (8) **`coderef`** — resolve external code snippet inclusions. (9) **`chunk`** — execute topic split/combine decisions based on `@chunk` attributes. (10) **`maplink`** — generate links from relationship tables. (11) **`debug-filter`** — apply DITAVAL filtering with diagnostics. **Why separate targets:** (1) **Ordering** — each step depends on the previous (e.g., conref resolution needs keyrefs resolved first). (2) **Injection points** — plugins can hook before/after specific steps via extension points like `depend.preprocess.conref.pre`. (3) **Composability** — the pipeline can be reconfigured for different output formats (some formats skip certain steps). (4) **Debugging** — you can inspect the temp directory after each step to see exactly where processing went wrong. A monolithic process would prevent all four of these capabilities.

47. You need to add a custom build step that runs after conref resolution but before the HTML5 transform. How would you implement this as a plugin?

    > **Answer:** Create a plugin that uses the `depend.preprocess.post` extension point (which runs after all preprocessing, including conref resolution, but before the format-specific transform). **plugin.xml:** `<plugin id="com.yourco.custom-step" version="1.0.0"><require plugin="org.dita.html5"/><feature extension="depend.preprocess.post" value="custom-post-preprocess"/></plugin>`. Create a `build.xml` Ant file in the plugin directory defining the target: `<target name="custom-post-preprocess">` containing your custom processing logic (e.g., an XSLT transformation on the temp directory files, a validation script, or metadata injection). The Ant target has access to the preprocessed files in the temp directory, where all conrefs and keyrefs are already resolved. For more granular placement (specifically after conref but before other steps), you could use the legacy `depend.preprocess.conref.post` hook, but this is less stable across DITA-OT versions — the high-level `depend.preprocess.post` is the recommended approach.

48. Explain the XSLT import pattern for safe customization. Why is `<xsl:import>` safer than copying and modifying default stylesheets?

    > **Answer:** The XSLT import pattern works as follows: your plugin's override.xsl starts with `<xsl:import href="plugin:org.dita.html5:xsl/dita2html5.xsl"/>`, which imports all default templates. You then add only the specific templates you want to override — everything else falls through to the imported defaults. **Why this is safer than copy-paste:** (1) **Minimal surface area** — you maintain only 10 lines instead of 2,000. When DITA-OT is upgraded and the default templates change (bug fixes, new features), your import automatically picks up all the changes. Copy-paste means you must manually merge every upgrade. (2) **XSLT precedence rules** — templates in the importing stylesheet automatically win over imported ones, so your overrides take effect without needing to remove the originals. (3) **`xsl:next-match` layering** — you can wrap default behavior without replacing it: `<xsl:template match="*[contains(@class,' topic/table ')]"><div class="table-wrap"><xsl:next-match/></div></xsl:template>`. This adds a wrapper `<div>` while preserving all default table processing. Copy-paste would require maintaining the entire table template. (4) **Multiple plugins can coexist** — each imports and overrides different templates without conflicting.

49. Describe how you would containerize a DITA-OT build environment using Docker. What goes in the Dockerfile, and how do you handle custom plugins?

    > **Answer:** **Dockerfile:** Start from the official image: `FROM ghcr.io/dita-ot/dita-ot:4.3`. Copy custom plugins into the container: `COPY plugins/com.extense.html5.branded /opt/app/plugins/com.extense.html5.branded` and `COPY plugins/com.extense.pdf.branded /opt/app/plugins/com.extense.pdf.branded`. Run plugin integration: `RUN dita --install`. This bakes the plugins into the image so they don't need to be installed on every build. **Usage:** Mount your content directory and run: `docker run --rm -v "$PWD":/src ghcr.io/dita-ot/dita-ot:4.4 -i /src/dita/maps/book.ditamap -o /src/out/html5 -f html5 -v`. **Why Docker:** Consistent Java/toolchain version across all environments; predictable plugin install behavior; easy CI parity (local build == CI build); no "works on my machine" issues. **CI integration:** The same Docker image used locally is referenced in GitHub Actions: `container: { image: your-registry/dita-ot-custom:4.3 }`. **Multi-stage builds** can create separate images for different plugin sets or output format specializations.

50. How would you debug a plugin conflict where two plugins modify the same extension point?

    > **Answer:** **Step 1 — Identify the conflict:** Run `dita --plugins` to list all installed plugins and their declared extension points. Look for two plugins both declaring `<feature extension="dita.xsl.html5">` or similar. Check the DITA-OT integration log during `dita --install` for warnings about conflicting contributions. **Step 2 — Understand precedence:** When two plugins extend the same XSLT extension point, both stylesheets are imported. XSLT import precedence determines which templates win — the last imported stylesheet has higher precedence. Plugin load order depends on `<require>` declarations. **Step 3 — Inspect temp directory:** Build with `--debug` and examine the generated XSLT in the temp directory to see the effective combined stylesheet. Check which template is actually matching for the element in question. **Step 4 — Resolve:** Options include: (a) add a `<require>` dependency from one plugin on the other to control load order; (b) use `xsl:next-match` in both plugins so they layer rather than conflict; (c) merge the two plugins into one with a single coherent override; (d) use more specific template match patterns so each plugin handles different elements. **Step 5 — Build in strict mode** (`--processing-mode=strict`) to surface warnings early.

## Part VIII — XSLT Customization

### Intermediate

51. What is the DITA `@class` attribute, and why is matching on `@class` preferred over matching on element names in XSLT?

    > **Answer:** The `@class` attribute encodes the specialization ancestry of every DITA element — e.g., a `<step>` element has `class=" topic/li task/step "`. This means a `<step>` is a specialization of `<li>` in the topic domain. **Why match on `@class`:** If you write a template matching the element name `step`, it only matches `<step>` elements. But if someone creates a specialization of `<step>` (e.g., `<substep>`), your template won't apply. Matching on `@class` with `*[contains(@class,' task/step ')]` matches both `<step>` and any specialization of it, because all specializations inherit the base class string. This makes XSLT customizations **specialization-safe** — they work with any domain or structural specialization without modification. The canonical pattern includes a space before and after the class token to avoid partial matches: `contains(@class,' topic/p ')` won't accidentally match `topic/ph`.

52. Explain the `xsl:next-match` pattern. How does it enable layered customization without duplicating default logic?

    > **Answer:** `xsl:next-match` tells the XSLT processor: "After my template runs, continue processing with the next matching template in import precedence order." This enables a **wrapper pattern** where you add custom markup around the default output without replacing it. Example: `<xsl:template match="*[contains(@class,' topic/table ')]"><div class="table-wrap"><xsl:next-match/></div></xsl:template>`. This wraps every table in a `<div class="table-wrap">` while preserving all default table rendering logic. Without `xsl:next-match`, you'd need to copy the entire default table template (potentially 50+ lines) and paste it inside your wrapper. With the pattern, you maintain 3 lines. This is particularly powerful for: adding CSS hook classes, injecting wrapper `<div>`s for layout, adding `data-` attributes for JavaScript interaction, and inserting accessibility attributes (`aria-label`). Multiple plugins can each use `xsl:next-match` on the same element, creating a layered chain of customizations.

53. You want to inject a custom `<meta>` tag into the `<head>` of every HTML5 page. Which named template would you override, and how?

    > **Answer:** Override the `gen-user-head` named template. This is a hook specifically designed for injecting content into the HTML `<head>` section. In your plugin's override.xsl: `<xsl:import href="plugin:org.dita.html5:xsl/dita2html5.xsl"/>` then: `<xsl:template name="gen-user-head"><meta name="generator" content="DITA-OT Custom"/><meta name="audience" content="{//prolog/metadata/audience/@type}"/><xsl:next-match/></xsl:template>`. This injects custom `<meta>` tags while preserving any other head content from the default template via `xsl:next-match`. You can also pull dynamic values from the topic's prolog metadata using XPath — for example, extracting audience, keywords, or custom `<data>` values and rendering them as `<meta>` tags for SEO or analytics purposes. Register the override XSLT via the `dita.xsl.html5` extension point in your plugin.xml.

54. Describe the three phases of the DITA-OT HTML5 transform. At which phase would you customize sidebar navigation?

56. You need to add anchor IDs and deep-linking to every `<step>` in task topics. Describe the XSLT template match pattern and the generated HTML structure.

This matches `<step>` elements and any specialization of steps, using the class attribute for specialization safety. **Generated HTML:** `<li class="dita-step" id="step-{position()}" data-step="{position()}"><a href="#step-{position()}" class="step-anchor" aria-label="Step {position()}">#</a><xsl:next-match/></li>`. 57. Compare Strategy A (plugin + `<xsl:import>`) vs. Strategy B (`args.xsl` parameter) for XSLT customization. When would you use each?

    > **Answer:** **Strategy A — Plugin + `<xsl:import>` (recommended):** Create a plugin with override.xsl that imports defaults and overrides specific templates. Register via `dita.xsl.html5` extension point. Installed once with `dita --install`. **Advantages:** Upgrade-safe, portable, composable with other plugins, supports CSS/JS bundling, proper dependency management via `<require>`. **Strategy B — `args.xsl` parameter:** Pass a custom XSLT file directly on the command line: `dita -i map.ditamap -f html5 --args.xsl=path/to/custom.xsl`. No plugin installation required. **Advantages:** Quick prototyping, no need to manage plugin lifecycle, easy to test one-off changes. **Tradeoffs:** `args.xsl` skips the plugin integration layer, so it may not compose well with other plugins; it's not portable (path must be correct on every machine); it cannot bundle CSS/JS; and it requires passing the parameter on every build. **When to use Strategy A:** Production customizations, team-shared overrides, CI/CD pipelines, anything that needs to survive DITA-OT upgrades. **When to use Strategy B:** Quick experiments, debugging, temporary overrides during development.

58. How would you implement a collapsible sidebar TOC using the `dita.xsl.html5.toc` extension point? What role does JavaScript play?

    > **Answer:** **Plugin setup:** In plugin.xml, declare: `<feature extension="dita.xsl.html5.toc" file="xsl/toc/toc-override.xsl"/>`. **XSLT override:** Import the default TOC stylesheet: `<xsl:import href="plugin:org.dita.html5:xsl/toc.xsl"/>`. 59. A client reports that their XSLT override works in DITA-OT 3.x but breaks in 4.x. What is your debugging workflow, and what are the most likely causes?

    > **Answer:** **Debugging workflow:** (1) Build with `--processing-mode=strict -d` to get detailed debug output. (2) Compare the temp directory output between 3.x and 4.x builds — look for differences in the preprocessed intermediate files. (3) Find the template that's failing — search the DITA-OT 4.x source XSLT for the element class your template matches. (4) Compare the default template signatures between versions. **Most likely causes:** (1) **`preprocess2` (map-first) replacing `preprocess`** — HTML5 uses map-first preprocessing as of 4.2. Extension points that hooked into legacy preprocessing steps (e.g., `depend.preprocess.conref.pre`) may not fire. Fix: use high-level hooks (`depend.preprocess.pre`/`post`). (2) **Changed import paths** — the internal XSLT file structure may have been reorganized. Fix: update `<xsl:import>` paths and use the `plugin:` URI scheme (`plugin:org.dita.html5:xsl/dita2html5.xsl`) which is version-stable. (3) **XSLT version change** — DITA-OT 4.x uses XSLT 3.0 with Saxon. Templates using deprecated XSLT 1.0 patterns may behave differently. (4) **Changed default template behavior** — the default template your override relied on (via `xsl:next-match`) may have changed signature or output. **Upgrade strategy:** Build with no customization changes, compare outputs, adjust imports, minimize template rewrites, and maintain a plugin CHANGELOG.

## Part IX — AI Pipeline & RAG Integration

### Intermediate

60. What is Retrieval-Augmented Generation (RAG), and what are its four stages (ingest, retrieve, rerank, generate)?

    > **Answer:** RAG is a pattern that enhances a large language model (LLM) by providing it with relevant, retrieved context from a knowledge base rather than relying solely on the model's training data. The four stages are: **(1) Ingest** — content is chunked, embedded as vectors, and stored in a vector database along with metadata. **(2) Retrieve** — a user query is embedded and compared against stored vectors via cosine similarity to find the top-k most relevant chunks. **(3) Rerank** — a cross-encoder reranking model scores the retrieved chunks more precisely, filtering out false positives and reordering by relevance. **(4) Generate** — the top reranked chunks are injected into the LLM prompt as context, and the model generates an answer grounded in that content. RAG eliminates hallucination by grounding answers in actual documentation, and it keeps answers up-to-date without retraining the model.

61. Why is DITA an ideal source format for RAG pipelines compared to unstructured text?

    > **Answer:** DITA provides **structural semantics** that unstructured formats lack: (1) **Information typing** — concept, task, and reference types enable the pipeline to classify content by intent (explanatory vs. procedural vs. lookup). (2) **Section IDs** — `<section id="...">` provides natural chunk boundaries that align with semantic meaning, avoiding mid-sentence splits. (3) **Rich metadata** — `<prolog>` contains audience, category, keyword, and custom `<data>` elements that can be indexed as vector metadata for filtered search. (4) **Reuse mechanics** — conref/keyref ensure a single source of truth, so the pipeline never ingests conflicting duplicate content. (5) **Consistent structure** — every topic follows a schema, making parsing deterministic. Unstructured text (Word, Markdown) requires heuristic parsing that is fragile and loses semantic context. DITA's structure maps directly to RAG pipeline requirements: clean chunks, filterable metadata, typed content, and traceable citations.

62. What is section-aware chunking? How does it differ from naive text splitting, and why does it produce better retrieval results?

    > **Answer:** **Naive text splitting** divides content by fixed token count (e.g., every 500 tokens) regardless of meaning boundaries. This produces chunks that start mid-sentence, split code examples, and separate headings from their content. **Section-aware chunking** uses DITA's structural elements (`<section>`, `<body>`, `<steps>`) as chunk boundaries. Each chunk corresponds to a semantically complete unit — a full section, a complete task procedure, or a coherent concept explanation. Parameters: minimum 100 tokens (avoid fragments), maximum 1,500 tokens (avoid diluted embeddings), 2-sentence overlap between adjacent chunks for context continuity. **Why better retrieval:** Semantic chunks produce more focused embeddings because the vector represents a single coherent idea instead of a fragment spanning two unrelated topics. This increases cosine similarity scores for relevant matches and reduces false positives. Section-aware chunking improved retrievable chunk count from 80 (naive) to 210+ in the training case study.

### Advanced

63. Explain how DITA metadata enables filtered vector search. Give an example of a multi-faceted query that filters by audience, difficulty, and content domain simultaneously.

    > **Answer:** When DITA topics are ingested into a vector database, the `<prolog>` metadata is stored as **filterable fields** alongside the embedding vector. This enables **pre-filtering** — the vector search is restricted to only chunks matching the metadata criteria before computing cosine similarity, dramatically reducing noise. **Example multi-faceted query:** A beginner developer asks the chatbot: "How do I set up my first DITA build?" The query is embedded AND the vector search filter is set to: `audience="developer" AND difficulty="beginner" AND category="publishing"`. This eliminates advanced XSLT customization chunks, writer-audience content, and architecture topics from the candidate set before any similarity scoring. Without metadata filtering, the top-k results might include advanced plugin development content that happens to share embedding similarity with "build." DITA metadata fields commonly indexed: `audience/@type`, `category`, custom `<data name="difficulty">`, `<data name="chatbot-priority">`, `prodinfo/prodname`, and `keywords/keyword`.

64. You're designing a RAG pipeline for a DITA-based knowledge base. Describe your chunking strategy, embedding model choice, retrieval parameters (top-k, score threshold), and reranking approach.

    > **Answer:** **Chunking strategy:** Section-aware chunking using `<section>` and `<body>` boundaries. Min 100 tokens, max 1,500 tokens, 2-sentence overlap. Preserve topic_id and section_id in chunk metadata. Include parent topic title as context prefix in each chunk. **Embedding model:** `text-embedding-3-large` with 3,072 dimensions. Cosine similarity metric. This model handles technical content well and the high dimensionality captures nuanced distinctions between similar DITA concepts. **Retrieval parameters:** `top_k=8` (fetch 8 candidates from vector search), `score_threshold=0.72` (discard chunks below this cosine similarity). Metadata pre-filter by audience and product when available from the user profile. **Reranking:** Apply a cross-encoder reranker (e.g., `cross-encoder/ms-marco-MiniLM-L-12-v2`) to the top-k results. The cross-encoder scores query-chunk pairs jointly, achieving 18-24% higher accuracy than vector similarity alone. After reranking, take the top 3-4 chunks as context for the LLM prompt. **Priority weighting:** Apply `chatbot-priority` boost multipliers: high=1.3x, medium=1.0x, low=0.8x to the similarity scores before reranking.

65. How does the `chatbot-priority` metadata field affect retrieval ranking? What is the role of cross-encoder reranking vs. vector similarity?

    > **Answer:** **`chatbot-priority`** is a custom DITA `<data>` element in the prolog (e.g., `<data name="chatbot-priority" value="high"/>`) that signals how important a chunk is for chatbot answers. During retrieval, the raw cosine similarity score is multiplied by a boost factor: `high` = 1.3x (overview/getting-started content prioritized), `medium` = 1.0x (no change), `low` = 0.8x (reference/appendix content deprioritized). This means a "high" priority chunk with a 0.78 similarity score becomes 1.014, outranking a "medium" chunk at 0.82. **Vector similarity vs. cross-encoder reranking:** Vector similarity is a fast, approximate relevance signal — it compares independently-embedded query and chunk vectors. It handles scale (millions of chunks) but misses nuance. **Cross-encoder reranking** takes the top-k vector results and re-scores them by processing the query and each chunk together through a transformer model. This joint processing captures subtle relevance signals (negation, context dependency, synonyms) that independent embeddings miss. The cross-encoder improves retrieval accuracy by 18-24% but is too slow to run over the full corpus — hence the two-stage pipeline: fast vector retrieval → precise cross-encoder reranking.

66. How would you implement citation generation that links chatbot answers back to specific DITA sections using `<section id>` and `<resourceid>`?

    > **Answer:** **At ingest time:** When chunking DITA topics, store `topic_id` (from the topic's `@id` attribute) and `section_id` (from `<section id="...">`) in the vector database metadata for each chunk. Also store `resourceid` if the topic has `<resourceid appid="..." ux-source-priority="topic-and-map-first"/>`. **At generation time:** After the LLM generates an answer using retrieved chunks, extract the metadata from each chunk that contributed to the answer. Format citations as: `[topic_id § section_id]` — e.g., `[rag-pipeline § chunking-strategy]`. **Linking:** If the published HTML5 output is available online, construct deep links using the topic filename and section anchor: `https://docs.example.com/rag-pipeline.html#chunking-strategy`. The `resourceid` can map to an external CMS URL or API endpoint for systems that don't use file-based URLs. **Display:** Append citations at the end of the chatbot response: "Sources: [RAG Pipeline § Chunking Strategy](rag-pipeline.html#chunking-strategy), [Embedding Config § Model Selection](embedding-config.html#model-selection)". This provides **traceability** — users can verify the answer against the source documentation.

67. Describe how DITA information typing enables intent routing in a chatbot — e.g., routing "What is X?" to concept topics and "How do I X?" to task topics.

    > **Answer:** DITA's three core information types map directly to user intent patterns: **Concept topics** answer "What is X?" / "Explain X" / "Why does X..." — they contain definitions, overviews, and explanatory content. **Task topics** answer "How do I X?" / "Steps to X" / "Set up X" — they contain sequential procedures with `<steps>`. **Reference topics** answer "What are the options for X?" / "List of X" / "API parameters for X" — they contain tables, property lists, and specifications. **Implementation:** (1) Store `information_type` (concept/task/reference) as metadata in the vector database. (2) Classify the user's query intent using NLP or simple pattern matching (question words map to types). (3) Apply a metadata pre-filter matching the detected intent to the information type, or boost the similarity score for matching types. **Example:** Query: "How do I install DITA-OT?" → Intent: procedural → filter/boost `information_type="task"` → retrieve the installation task topic → return step-by-step instructions. Without intent routing, the concept topic explaining "what DITA-OT is" might rank equally with the task topic, diluting the answer quality.

74. **Debugging scenario:** A DITA build succeeds but the PDF output is missing three topics that appear in the HTML5 output. Where do you start troubleshooting?

    > **Answer:** **Step 1 — Check DITAVAL filtering:** If different DITAVAL files or profiles are used for PDF vs. HTML5 builds, the missing topics may be filtered out by `<prop att="audience" val="..." action="exclude"/>` in the PDF DITAVAL. Compare the DITAVAL files used for each build. **Step 2 — Check the ditamap:** Look for `@print="no"` or `@format` attributes on the `<topicref>` entries for the missing topics. `print="no"` explicitly excludes content from print (PDF) output. **Step 3 — Check `@processing-role`:** If topicrefs have `processing-role="resource-only"`, they're available for conref/keyref resolution but not rendered in output. **Step 4 — Check the temp directory:** Build with `-d -t temp-debug` and examine the preprocessed `.dita` files in the temp folder. If the topics are present in temp but missing from output, the issue is in the PDF transform. If they're missing from temp, the issue is in preprocessing (filtering/chunking). **Step 5 — Check `@chunk` attributes:** Incorrect chunking can merge or split topics unexpectedly in PDF. **Step 6 — Check PDF plugin bookmap handling:** If using a `<bookmap>`, the PDF plugin processes `<chapter>`/`<appendix>` elements differently than `<topicref>`. Topics not wrapped in a book-level container may be silently dropped. **Step 7 — Run with `--processing-mode=strict`** to surface any warnings about unresolved references or skipped topics.

75. **Architecture scenario:** Your team is documenting a SaaS product with three editions (Starter, Professional, Enterprise). Each edition shares 70% of the same content but has unique features. Design the DITA information architecture — maps, keys, DITAVAL filters, and reuse strategy.

    > **Answer:** **Map architecture:** One root map per edition: `starter-guide.ditamap`, `professional-guide.ditamap`, `enterprise-guide.ditamap`. All three import a shared `common-content.ditamap` submap containing the 70% shared topics. Edition-specific topics live in separate submaps: `starter-features.ditamap`, `pro-features.ditamap`, `enterprise-features.ditamap`. **Key-based variable reuse:** Define a key space in each root map for edition-specific values: `<keydef keys="product-name"><topicmeta><keywords><keyword>Professional Edition</keyword></keywords></topicmeta></keydef>`, `<keydef keys="max-users"><topicmeta><keywords><keyword>50</keyword></keywords></topicmeta></keydef>`. Topics reference `<ph keyref="product-name"/>` and `<ph keyref="max-users"/>` — values resolve per-edition at build time. **DITAVAL filtering:** Three DITAVAL files: `starter.ditaval` excludes `@product="professional"` and `@product="enterprise"` content; similarly for each edition. Shared topics use `@product` attributes on elements that differ: `<p product="enterprise">SSO is available with Enterprise.</p>`. **Reuse strategy:** Common procedures (login, navigation, settings) are authored once in shared topics. Edition-specific steps within shared tasks use conditional `@product` attributes. Feature comparison tables use conkeyref to pull edition-specific cells. **Build:** CI/CD builds all three editions in parallel: `dita -i starter-guide.ditamap -f html5 --filter=starter.ditaval`, etc.

76. **Plugin development scenario:** The marketing team wants every HTML5 page to include Open Graph `<meta>` tags pulled from DITA prolog metadata. How would you implement this as a DITA-OT plugin?

    > **Answer:** **Plugin structure:** `com.yourco.html5.opengraph/` with `plugin.xml`, `xsl/opengraph.xsl`. **plugin.xml:** `<plugin id="com.yourco.html5.opengraph" version="1.0.0"><require plugin="org.dita.html5"/><feature extension="dita.xsl.html5" file="xsl/opengraph.xsl"/></plugin>`. **opengraph.xsl:** Import defaults and override the `gen-user-head` named template: `<xsl:import href="plugin:org.dita.html5:xsl/dita2html5.xsl"/>`, then `<xsl:template name="gen-user-head"><meta property="og:title" content="{//*[contains(@class,' topic/title ')][1]}"/><meta property="og:description" content="{//*[contains(@class,' topic/shortdesc ')][1]}"/><meta property="og:type" content="article"/><xsl:if test="//prolog/metadata/data[@name='og-image']"><meta property="og:image" content="{//prolog/metadata/data[@name='og-image']/@value}"/></xsl:if><xsl:next-match/></xsl:template>`. This pulls `og:title` from the topic title, `og:description` from shortdesc, and `og:image` from a custom `<data name="og-image" value="https://...">` element in the prolog. **Install and test:** `dita --install plugins/com.yourco.html5.opengraph`, then build. Verify by checking `<head>` of generated HTML files for the Open Graph meta tags.

77. **AI integration scenario:** Your chatbot is returning irrelevant answers to beginner-level user questions because it retrieves advanced expert content. How would you use DITA metadata to fix this?

    > **Answer:** **Root cause:** The vector search doesn't account for difficulty level — advanced topics with high semantic similarity to beginner queries rank equally. **Fix using DITA metadata:** (1) **Add difficulty metadata** to every topic prolog: `<data name="difficulty" value="beginner"/>`, `<data name="difficulty" value="advanced"/>`. (2) **Index difficulty as metadata** in the vector database so it's available as a filter field. (3) **Detect user level** — from user profile settings, conversation history, or explicit query signals ("I'm new to...", "getting started with..."). (4) **Apply metadata pre-filter** to the vector search: when a beginner user queries, set filter `difficulty IN ["beginner", "intermediate"]` to exclude advanced content before similarity scoring. (5) **Adjust chatbot-priority** — set `<data name="chatbot-priority" value="high"/>` on getting-started and overview topics. The 1.3x boost multiplier ensures these rank above niche advanced topics even when vector similarity is close. (6) **Fallback strategy:** If the filtered search returns too few results (below threshold), progressively relax the filter to include higher difficulty levels. This layered approach uses DITA's structured metadata to implement what would be impossible with unstructured content.

78. **Scale scenario:** Your DITA content set has grown to 2,000 topics across 15 products. Builds are slow and the team struggles with key conflicts. What architectural changes would you propose?

    > **Answer:** **Key conflict resolution:** (1) Implement **key scoping** — each product's submap defines its own key scope: `<mapref href="product-a/product-a.ditamap" keyscope="prodA"/>`. References use scoped keys: `<ph keyref="prodA.product-name"/>`. This eliminates cross-product key collisions. (2) Establish key naming conventions: `{scope}.{category}.{name}` (e.g., `prodA.ui.save-button`). **Build performance:** (1) Use **DITA-OT project files** (`.xml`) to define multi-deliverable builds and enable parallel processing. (2) Enable `preprocess2` (map-first preprocessing) which is more efficient for large map hierarchies. (3) Containerize builds with Docker and run product builds in parallel on CI/CD (separate GitHub Actions jobs per product). (4) Implement **incremental builds** — only rebuild products whose source files changed (use git diff to detect changed submaps). (5) Increase JVM heap: `-Xmx4g` for large builds. **Architecture:** (1) Separate shared content into a `common/` library with its own submap, referenced by all products. (2) Use a **root supermap** for cross-product builds and individual product root maps for single-product builds. (3) Implement a subject scheme at the organization level to standardize taxonomy across all 15 products. **Governance:** Automated validation in CI (schematron rules for required metadata, ID format, key naming).

79. **Quality scenario:** A new technical writer joins your team and starts creating DITA topics with inconsistent IDs, missing metadata, and mixed content types in single topics. Design a governance process and tooling strategy to prevent this.

    > **Answer:** **Authoring standards:** (1) Create a DITA style guide defining: ID format conventions (`{type}-{module}-{name}`, e.g., `task-install-dita-ot`), required prolog elements (audience, category, keywords, difficulty), one information type per topic (never mix concept and task content). (2) Provide topic templates in the XML editor (Oxygen/XMetal) with pre-populated prolog skeletons. **Automated validation:** (1) **Schematron rules** — create custom Schematron that validates: `@id` matches the naming pattern, prolog contains required elements, topic type matches content (a concept topic must not contain `<steps>`). Run Schematron as a pre-commit hook. (2) **CI/CD gate** — add a validation step in GitHub Actions that runs `dita --input=map.ditamap --format=html5 --processing-mode=strict` plus custom Schematron checks. Fail the build on violations. (3) **Editor configuration** — configure Oxygen Author with custom actions/templates that auto-generate conforming IDs and pre-fill prolog fields. **Process:** (1) Mandatory peer review for all new topics (PR-based workflow). (2) Bi-weekly content audit script that reports metadata coverage and ID conformance percentages. (3) Onboarding checklist with hands-on exercises covering DITA authoring standards. (4) Gradual enforcement — warn for the first month, block non-conforming commits after.

80. **Localization scenario:** Your DITA content needs to be translated into 8 languages. How does DITA's structure (conref, keyref, filtering) support or complicate the localization workflow?

    > **Answer:** **How DITA supports localization:** (1) **Conref reduces translation volume** — content reused via conref is translated once at the source topic. All references automatically pick up the translated version. If you have 30% conref reuse, you translate 30% less content. (2) **Keyref enables locale-specific values** — define locale-specific key definitions (product names, legal terms, UI labels) in per-language key maps. The same topic source uses `<ph keyref="product-name"/>` and resolves to the correct translation at build time. (3) **DITAVAL filtering** — region-specific content (US vs. EU regulatory text) can be filtered using `@audience` or `@props` attributes. Each locale build applies the appropriate DITAVAL. (4) **Structured XML enables TM leverage** — translation memory tools (SDL Trados, memoQ) handle DITA natively, achieving high TM leverage due to consistent segment structure. **Complications:** (1) **Conref target locking** — translators must not modify conref target files without understanding the impact on all referencing topics. Translation tools must be configured to protect conref sources. (2) **Key scope complexity** — nested key scopes with locale overrides can create confusing resolution chains. (3) **Inline elements** — heavy use of `<ph>`, `<uicontrol>`, and other inline elements creates short, fragmented segments that reduce TM leverage and complicate word counting. (4) **Bi-directional text** — Arabic/Hebrew require RTL support in both authoring and publishing. The HTML5 plugin must handle `dir="rtl"` attributes. **Best practice:** Use XLIFF-based round-trip: export DITA to XLIFF, translate, re-import. This preserves all markup and enables diff-based re-translation when source content changes.

## Quick-Fire / Definitions

81. What is a `conref`?

    > **Answer:** A content reference (`conref`) is DITA's mechanism for reusing content by reference. You point to a specific element in another topic using `conref="topic.dita#topicid/elementid"`, and at build time, the referenced content replaces the referencing element. It's like a transclusion — author once, reuse everywhere. Changes to the source automatically propagate to all conref consumers.

82. What is a `keyref`?

    > **Answer:** A `keyref` is an indirect reference resolved through a key defined in the ditamap. Instead of hard-coding a file path, you reference a key name (`<ph keyref="product-name"/>`), and the map's `<keydef>` resolves it to a value or target. This enables variable text, redirectable links, and context-dependent content — the same topic can display different values depending on which map builds it.

83. What is a DITAVAL file?

    > **Answer:** A DITAVAL file is an XML file that defines conditional processing rules — which content to include, exclude, or flag based on attribute values. Example: `<prop att="audience" val="admin" action="include"/>`, `<prop att="audience" val="developer" action="exclude"/>`. Applied at build time with `--filter=myfile.ditaval`, it enables single-source publishing of audience-specific or product-specific variants from one content set.

84. What is a subject scheme?

    > **Answer:** A subject scheme is a specialized DITA map that defines a controlled taxonomy — the allowed values for attributes like `@audience`, `@product`, and `@platform`. It replaces free-text attribute values with an enumerated, hierarchical vocabulary. This enforces consistency across authors and enables validation: if a writer uses `audience="devloper"` (typo), the schema-aware editor flags it as invalid.

85. What is the difference between `<topichead>` and `<topicgroup>`?

    > **Answer:** `<topichead>` creates a **titled grouping** in navigation — it generates a heading in the TOC (e.g., "Getting Started") without linking to a topic file. `<topicgroup>` creates an **untitled grouping** — it organizes topicrefs for authoring convenience or metadata inheritance but produces no visible heading in the output. Use `<topichead>` when you want a navigation label; use `<topicgroup>` when you want invisible structural organization.

86. What does `@chunk="to-content"` do on a `<topicref>`?

    > **Answer:** `chunk="to-content"` tells the DITA-OT to merge the referenced topic and all its child topicrefs into a **single output file**. Without chunking, each topicref produces a separate HTML page. With `chunk="to-content"`, the parent and all children are combined into one continuous page. This is useful for creating single-page guides, printable views, or reducing the number of output files.

87. What is the purpose of `<shortdesc>` in a DITA topic?

    > **Answer:** `<shortdesc>` is a brief description (1-2 sentences) that appears immediately after the topic title. It serves multiple purposes: (1) provides a summary preview in link hover-overs and search results, (2) appears as the first paragraph of the topic body, (3) is used as the `<meta name="description">` in HTML output for SEO, (4) can be pulled into map-level link previews. Writing effective shortdescs is critical for content findability and user scanning.

88. What is specialization in DITA?

    > **Answer:** Specialization is DITA's extension mechanism for creating new element types and topic types by constraining existing ones. For example, `<task>` is a specialization of `<topic>` — it adds `<steps>`, `<prereq>`, and `<result>` while inheriting all base topic features. The `@class` attribute records the inheritance chain, ensuring that any tool processing the base type automatically handles the specialization. This enables domain-specific vocabularies (e.g., `<apiOperation>` specializing `<reference>`) without breaking existing toolchains.

89. What is the role of the `<prolog>` element?

    > **Answer:** `<prolog>` is the metadata container in every DITA topic, appearing between the title and body. It holds: `<author>`, `<critdates>` (created/revised dates), `<metadata>` (containing `<audience>`, `<category>`, `<keywords>`, and custom `<data>` elements), `<permissions>`, and `<resourceid>`. The prolog is not rendered in output by default but drives conditional processing, search indexing, metadata injection (HTML `<meta>` tags), RAG pipeline filtering, and content governance validation.

90. What is an extension point in DITA-OT?

    > **Answer:** An extension point is a named hook in the DITA-OT processing pipeline where plugins can inject custom behavior. Examples: `dita.xsl.html5` (add XSLT overrides to HTML5 transform), `dita.xsl.html5.toc` (customize TOC generation), `depend.preprocess.post` (run custom Ant targets after preprocessing), `dita.conductor.html5.param` (add custom build parameters). Plugins declare extensions in `plugin.xml` using `<feature extension="extension-point-name" file="custom-file"/>`. This architecture enables modular, composable customizations that don't modify DITA-OT core files.
