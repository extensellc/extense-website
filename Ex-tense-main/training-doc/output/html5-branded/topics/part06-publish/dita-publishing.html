Publishing with DITA-OT

What publishing means in DITA

DITA is a source format, not an output. Publishing is the process of:

  • resolving references (conref/keyref),
  • applying filtering rules,
  • transforming XML into HTML/PDF/other formats,
  • applying styling and navigation rules,
  • producing deliverables that match your product and brand.

Many DITA programs use the DITA Open Toolkit (DITA‑OT) as the publishing engine. Others use CCMS publishing services that may internally rely on DITA‑OT or custom pipelines.

This topic focuses on practical publishing with DITA‑OT because it is widely used and teaches you the core concepts that apply everywhere.

Inputs and outputs: what DITA-OT expects

Input

DITA‑OT publishes from a map (most common) or a single topic. In professional workflows:

  • maps are preferred because they define structure, keys, and context.

Output formats (common)

  • HTML5: website-like output with navigation
  • PDF: often XSL‑FO based (toolchain-dependent)
  • Other outputs via plugins or extensions

The key idea

A map is not just a TOC; it is the publication definition. If you publish the same topic under two different maps, you can get different results because:

  • key definitions differ,
  • filtering differs,
  • navigation context differs.

The “build pipeline” steps (what happens during publishing)

Publishing is more than “transform XML to HTML.”

A typical pipeline includes:

  1. Preprocess — parse the map, resolve topicrefs, collect key definitions, gather referenced topics and resources.
  2. Reference resolution — apply conref pulls, resolve keyrefs, validate IDs and targets.
  3. Filtering — include/exclude content based on rules, adjust link targets accordingly.
  4. Transform — generate intermediate or final formats, apply templates and rules for HTML/PDF.
  5. Postprocess — create navigation and search index assets, copy images/media, apply final styling.

Understanding these stages helps you debug issues:

  • missing content (filtering),
  • broken links (IDs, key spaces),
  • wrong product name (keys),
  • missing images (resource paths).

Basic DITA-OT commands (the 80/20)

Publish HTML5

dita -i maps/guides/getting-started.ditamap -f html5 -o out/html

Publish PDF (if configured)

dita -i maps/guides/getting-started.ditamap -f pdf -o out/pdf

Practical tips

  • Always use an explicit output directory (-o).
  • Keep your build artifacts out of your source tree where possible.
  • Use a consistent build script so builds are repeatable.

Repeatable builds: project files and build scripts

As soon as you have:

  • multiple deliverables,
  • filtering variants,
  • consistent parameters,

…you should formalize builds.

Why repeatability matters

Repeatable builds ensure:

  • anyone can build the same output,
  • CI builds match local builds,
  • releases are reproducible.

A typical approach

  • Create a build script (build.sh or build.ps1) that runs defined publications.
  • Use DITA‑OT project files (where supported in your version) or a wrapper configuration.

Even without deep customization, a script with consistent parameters eliminates “works on my machine” issues.

Parameters: controlling output behavior

DITA‑OT allows parameters to control:

  • toc depth,
  • chunking behavior,
  • navigation,
  • filtering integration,
  • output artifacts.

In practice, you’ll maintain:

  • a baseline parameter set for HTML,
  • a baseline parameter set for PDF,
  • variant overrides per product/platform.

Common DITA-OT parameters (all transtypes)

These parameters work across most output formats:

Parameter Values / Example What it does
-i / --input maps/book.ditamap Input map or topic file
-f / --format html5, pdf, xhtml Output transtype
-o / --output out/html5 Output directory
--filter build/ditaval/web.ditaval DITAVAL filter file for conditional processing
--propertyfile build/html5.properties Load parameters from a properties file
--processing-mode strict, lax, skip How to handle errors (strict = fail on warnings)
--temp /tmp/dita-temp Temporary directory (useful for debugging)
-v / --verbose (flag) Enable verbose logging
--debug (flag) Enable debug logging (very detailed)

HTML5-specific parameters

Parameter Values / Example What it does
--nav-toc none, partial, full Table of contents generation in each page
--args.copycss yes, no Copy custom CSS to output directory
--args.css brand.css Name of custom CSS file to link
--args.cssroot css/ Source directory for CSS files
--args.csspath css Path in output where CSS is placed
--args.hdr header.xml Custom header file for every page
--args.ftr footer.xml Custom footer file for every page
--args.gen.task.lbl YES, NO Generate step labels ("Step 1", "Step 2")
--args.rellinks none, all, nofamily Which related links to include
--args.outext .html Output file extension

PDF-specific parameters

Parameter Values / Example What it does
--args.bookmap-order discard, retain Keep or discard bookmap element order
--args.chapter.layout MINITOC, BASIC Chapter page layout style
--args.fo.userconfig fop.xconf FOP configuration file
--custom.xep.config xep.xml XEP formatter configuration
--pdf.version 1.5, 1.7 PDF version to generate

Using a properties file (recommended for teams)

Instead of long command lines, put stable settings in a .properties file:

# build/html5-sidebar.properties
nav-toc           = full
args.copycss      = yes
args.cssroot      = css/
args.css          = brand.css
args.gen.task.lbl = YES
args.rellinks     = nofamily

Then build with:

dita -i maps/book.ditamap -f html5 -o out/html5 \
  --propertyfile=build/html5-sidebar.properties \
  --filter=build/ditaval/web.ditaval

Example: passing parameters (Ant-style)

dita -i maps/guides/getting-started.ditamap -f html5 -o out/html \
  -Dargs.gen.task.lbl=true \
  -Dargs.rellinks=none
ℹ NOTE
Note: The -D prefix syntax is the Ant-style parameter passing. The --args.* syntax is recommended for DITA-OT 3.x+.

The exact parameters you use depend on your output requirements and your DITA-OT version.

Filtering in the build (DITAVAL concept)

Filtering is usually applied at build time by providing filtering rules.

Typical workflow:

  1. Tag content with attributes (platform="windows").
  2. Create a filter file that includes/excludes values.
  3. Run separate builds for each variant.

Example build strategy

  • build-windows-html
  • build-linux-html
  • build-windows-pdf
  • build-linux-pdf

Even if you later automate this, it’s valuable to think of variants as explicit publications.

Output customization: where to change what

HTML customization

Common customization areas:

  • CSS (theme),
  • HTML templates and headers/footers,
  • navigation behavior,
  • search integration (depending on output).

A practical approach:

  • start with CSS and minimal HTML template overrides,
  • avoid deep XSLT changes until necessary.

PDF customization

PDF often requires:

  • FO styling rules,
  • page masters, headers/footers,
  • tables and code block formatting,
  • numbering and cross-references.

PDF customization can become a deep specialization area. Start small:

  • change fonts, margins, headers,
  • adjust table styles,
  • make code blocks readable.

Then expand:

  • custom cover pages,
  • running headers and footers,
  • index generation and glossaries (if your pipeline supports them).

Plugin customization

DITA‑OT supports plugins to:

  • add new transforms,
  • override templates,
  • inject preprocessing steps,
  • create reports.

Use plugins when:

  • you need consistent custom behavior across builds,
  • you want to package customization cleanly,
  • parameter overrides aren’t enough.

Debugging build problems (a practical checklist)

Problem: Content missing

  • Check filtering rules.
  • Check map inclusion (topicref points to correct file).
  • Check if a topic is excluded by conditional processing attributes.
  • Check if content is pulled via conref and the target is missing.

Problem: Broken links or unresolved references

  • Validate IDs and references.
  • Ensure topic IDs and element IDs exist.
  • Check for renames that weren’t propagated.

Problem: Wrong product name or wrong URL

  • Check keys in the map.
  • Ensure you are building the correct map.
  • Check key scopes if advanced map structures exist.

Problem: Images missing

  • Ensure image paths are correct and copied into output.
  • Ensure images are referenced with correct relative paths.
  • Check whether filtering excludes topics that provide the images.

Problem: PDF formatting issues

  • Inspect FO output if accessible.
  • Identify whether the problem is content structure (tables, code blocks) or styling.
  • Apply minimal overrides first.

CI and automation: publishing like software

A mature DITA pipeline looks like software CI:

  • builds are triggered by commits,
  • validation runs automatically,
  • outputs are versioned and deployed.

Practical CI steps

  1. Validate XML (well-formed + schema/DTD as applicable).
  2. Run DITA‑OT build for each required publication.
  3. Fail if warnings exceed a threshold (ideally fail on unresolved refs).
  4. Publish artifacts to:

- documentation portal, - release package, - internal storage.

Output deployment patterns

  • Static site hosting for HTML outputs,
  • artifact repositories for PDFs,
  • CCMS publishing service for enterprise portals.

Professional publishing architecture (how doc teams scale)

As documentation grows, teams typically separate concerns:

  • Authoring repo (topics, maps, reuse, media)
  • Build system repo (pipeline scripts, configs)
  • Brand/theme repo (CSS, templates, PDF themes)
  • Deployment system (portal, CDN, internal hosting)

This separation:

  • keeps authors focused,
  • keeps customization controlled,
  • supports multiple brands/products.

A practical “starter publishing setup” (copyable blueprint)

Suggested folder layout

dita/
  maps/
  topics/
  reuse/
  media/
  ditaval/
  build/
    build.sh
    build.ps1
    params/
      html.properties
      pdf.properties

What goes where

  • maps/: deliverable definitions
  • topics/: content
  • reuse/: shared snippets
  • ditaval/: filtering rules per variant
  • build/: scripts + parameters

Build script responsibilities

  • pick input maps,
  • pick output formats,
  • apply filter rules,
  • pass consistent parameters,
  • place outputs in a clean directory.

Common publishing mistakes (and how to avoid them)

Mistake: Publishing topics directly instead of maps

Fix: publish maps so key spaces and navigation context are consistent.

Mistake: Customizing too early

Fix: start with minimal overrides; grow customization as requirements become clear.

Mistake: Not treating build warnings as serious

Fix: fail builds on unresolved references and broken conrefs; monitor warning counts.

Mistake: No repeatability

Fix: scripts/project files; document required DITA‑OT version and plugins.

Mistake: Mixing build outputs into source

Fix: separate out/ from source and keep it out of version control.
