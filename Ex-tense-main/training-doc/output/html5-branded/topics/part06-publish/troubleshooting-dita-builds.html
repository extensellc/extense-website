Troubleshooting DITA Builds

The debugging mindset

DITA-OT builds fail for predictable reasons. The key is reading the right logs, understanding which pipeline stage failed, and applying targeted fixes. This topic covers the most common failures and their solutions.

Verbose and debug logging

Always start diagnosis with verbose output:

# Verbose mode — shows warnings and processing steps
dita -i map.ditamap -f html5 -o out/ -v

# Debug mode — maximum detail (very verbose)
dita -i map.ditamap -f html5 -o out/ -d

# Save log to file for analysis
dita -i map.ditamap -f html5 -o out/ -v 2>&1 | tee build.log

Key log prefixes to watch for:

  • [WARN] — potential problems that do not stop the build but may cause missing content or broken links in the output.
  • [ERROR] — problems that prevent successful output generation.
  • [FATAL] — the build cannot continue at all.

Broken links and missing references

Symptom: Warnings like DOTJ013W or DOTX036W in the log — "file not found" or "target topic not found."

Common causes:

  • Wrong href path — relative paths are resolved from the map's directory, not the topic's directory. Double-check your directory structure.
  • Typo in filename or ID — DITA file references are case-sensitive on Linux/macOS.
  • Moved or renamed files — update all maps and conrefs that reference the file.
  • Missing keydef — a keyref points to a key not defined in the current map's key space.

Fix checklist:

  1. Grep for the broken reference: grep -r "filename.dita" maps/ submaps/
  2. Verify the file exists at the expected relative path.
  3. Check key definitions in the root map and submaps.
  4. Re-run with -v to confirm the warning is resolved.

Missing content in output

Symptom: Topics or sections that exist in your source do not appear in the output.

Common causes:

  • DITAVAL filtering — content is being excluded by a profiling rule. Check your --args.filter settings and profiling attributes.
  • Topic not referenced in map — a topic file exists but is not included in any <topicref>.
  • Broken conref — a conref target does not resolve, so the content silently disappears.
  • processing-role="resource-only" — the topicref is marked as resource-only (for key definitions) and is intentionally excluded from output.

Diagnosis:

# Check which topics were processed
dita -i map.ditamap -f html5 -o out/ -v 2>&1 | grep "Processing"

# Check filtering rules
grep -r "action=\"exclude\"" filters/

PDF-specific issues

Blank pages

  • FOP inserts blank pages to maintain recto/verso page numbering in double-sided layouts. This is intentional for print output.
  • To suppress: set force-page-count="auto" in your custom FO templates (requires plugin customization).

Missing or wrong fonts

  • FOP only uses fonts it knows about. Register custom fonts in fop.xconf:
<renderer mime="application/pdf">
  <fonts>
    <font metrics-url="fonts/OpenSans-Regular.xml"
          embed-url="fonts/OpenSans-Regular.ttf"
          kerning="yes">
      <font-triplet name="Open Sans" style="normal" weight="normal"/>
    </font>
  </fonts>
</renderer>

Apply with: --args.fo.userconfig=fop.xconf

Table layout overflow

  • Wide tables overflow the page. Use colspec with proportional widths (colwidth="1*") instead of fixed widths.
  • Reduce font size in table cells via custom FO attribute sets.

Image resolution

  • FOP renders images at their intrinsic resolution. For print, use images at 300 DPI minimum.
  • Set explicit width/height in <image> elements to control sizing.

Memory and performance issues

Symptom: java.lang.OutOfMemoryError: Java heap space

Fix: Increase the JVM heap size before running DITA-OT:

# Set maximum heap to 4 GB
export ANT_OPTS="-Xmx4096m"
dita -i large-map.ditamap -f pdf -o out/

For very large publications (500+ topics), consider:

  • Splitting into smaller maps built independently.
  • Using map-first preprocessing (preprocess2) which is more memory-efficient.
  • Running builds in Docker containers with controlled memory limits.

Plugin conflicts

Symptom: Build fails after installing a new plugin, or output looks wrong.

Diagnosis:

# List installed plugins
dita --plugins

# Check for duplicate template overrides
grep -r "xsl:template" plugins/*/xsl/ | grep "match="

Common issues:

  • Conflicting XSLT imports — two plugins override the same template. Check plugin.xml for dita.xsl.xhtml extension points.
  • Missing dependencies — a plugin requires another plugin that is not installed. Check <require> elements in plugin.xml.
  • Version mismatch — plugin was built for an older DITA-OT version. Check the <require> for org.dita.base version constraints.

Nuclear option: Reinstall DITA-OT from scratch and add plugins one at a time to isolate the conflict.

Common error codes quick reference

Table 1. Frequently seen DITA-OT error codes
Code Meaning Action
DOTX036W Target topic not available Fix href path or add missing file
DOTJ013W File not found Check relative path from map directory
DOTJ031E Conref target not found Verify conref path and target ID
DOTX050I Duplicate key definition Review key scopes in sub-maps
DOTA069W Key not resolved Add keydef to root map or check scope
PDFX004F FOP formatting error Check table widths, image paths, fonts

Systematic troubleshooting workflow

  1. Read the log — run with -v and search for [ERROR] and [WARN].
  2. Identify the stage — is the error in preprocessing (references), filtering, transformation, or postprocessing?
  3. Isolate — build a minimal map with just the problem topic to rule out map-level issues.
  4. Validate XML — run xmllint --noout topic.dita to check well-formedness.
  5. Check temp files — examine out/temp/ for intermediate files that show how DITA-OT processed your content.
  6. Compare — diff a working topic against the broken one to spot differences.
  7. Fix and re-run — make one change at a time and rebuild to confirm each fix.
