Authoring DITA Topics

What “authoring in DITA” really means

Writing DITA is not just writing in XML. It’s writing with:

  • a content model (what elements are allowed where),
  • a purpose (concept vs task vs reference),
  • a reuse strategy (what stays local vs shared),
  • a publishing strategy (HTML/PDF/portal),
  • a governance strategy (consistent attributes, IDs, terminology).

When DITA authoring goes wrong, it’s usually not because XML is hard. It’s because the team never agreed on:

  • topic size,
  • naming conventions,
  • metadata patterns,
  • reuse boundaries,
  • filtering rules.

This topic gives you a practical, “how real teams do it” guide.

Topic design: picking the right topic type

Concept topics (what/why)

Use concept topics to explain:

  • definitions and mental models,
  • background and rationale,
  • architectural overview,
  • decisions and tradeoffs,
  • “when to use” and “when not to use.”

A good concept topic:

  • answers a single question,
  • avoids procedural steps,
  • uses examples sparingly and purposefully.

Common elements:

  • <title>
  • <shortdesc> (optional but often valuable)
  • <conbody> with <p>, lists, sections as needed

Task topics (how)

Use tasks to instruct:

  • installation steps,
  • configuration procedures,
  • workflows and checklists,
  • troubleshooting actions (when they are step-based).

A good task topic:

  • begins with prerequisites and context,
  • has clear, testable steps,
  • distinguishes commands/UI actions,
  • ends with results or next steps.

Common elements:

  • <taskbody>
  • <prereq>, <context>, <steps>, <result>, <postreq>

Reference topics (facts/options)

Use reference topics for:

  • command syntax,
  • configuration parameters,
  • API endpoint specs,
  • field definitions,
  • error code tables.

A good reference topic:

  • is scannable,
  • uses tables consistently,
  • avoids long narrative explanations,
  • includes examples where needed.

The most important authoring decision: topic scope (“chunking”)

Why chunking matters

Chunking is how you decide what is:

  • one topic,
  • a section inside a topic,
  • a set of topics assembled by a map.

Chunking affects:

  • reuse,
  • translation cost,
  • search relevance,
  • link stability,
  • maintenance.

Practical chunking guidelines

Make a topic when:

  • the content solves one user goal,
  • the content can be reused elsewhere,
  • the content could appear in search results on its own.

Make a section (not a new topic) when:

  • the content is only meaningful in the same context,
  • splitting would create “orphan topics” with unclear purpose.

Split a topic if:

  • it has multiple distinct goals,
  • it becomes too long to scan,
  • it mixes concept and procedure heavily,
  • it gets reused partially (a sign you need smaller building blocks).

A simple heuristic

If a topic title needs “and” or “then” often, consider splitting:

  • “Install and Configure the System” → likely two tasks or a concept + task.
  • “What It Is and How to Use It” → concept + task.

IDs, filenames, and stability (treat your content like an API)

Why IDs matter more than you think

DITA references rely on IDs:

  • conref targets,
  • xrefs,
  • map references,
  • CCMS reuse references.

If IDs change frequently, you break:

  • reuse,
  • links,
  • downstream systems and portals.

Recommended patterns

File names

  • Use lowercase with hyphens: install-windows.dita
  • Keep names stable over time
  • Avoid “v2” or dates unless truly necessary

Topic IDs

  • Match the filename or a stable semantic label:

- file: install.dita - topic id: install

  • Avoid auto-generated IDs like topic_1234 if you can

Element IDs for reuse targets

  • Use meaningful IDs for reusable blocks:

- install-note - admin-warning - prereq-java

Stability rule

  • Treat changing an ID like changing a public API:

- do it rarely, - communicate it, - update dependent references, - validate builds.

Writing style in DITA (structure first, prose second)

DITA is not a prose-first system. It’s structure-first:

  • the structure communicates intent to both humans and machines,
  • the prose fills in the details.

Task writing best practices

  • Use imperative verbs: “Click”, “Select”, “Run”, “Verify”.
  • One action per step.
  • Put expected results in <result> or a final step.
  • Separate UI and command line procedures if they diverge significantly.
  • Use <uicontrol>, <menucascade>, <filepath>, <cmdname> when available (domain-dependent).

Concept writing best practices

  • Start with a crisp definition or summary.
  • Use short paragraphs.
  • Use examples and diagrams deliberately.
  • Avoid hidden tasks (“Do X, do Y”) unless it’s truly conceptual.

Reference writing best practices

  • Use consistent tables for parameters/options.
  • Use stable ordering (alphabetical or logical grouping).
  • Provide defaults, constraints, and examples.

Practical authoring templates (copy/paste patterns)

Concept template

<concept id="...">
  <title>...</title>
  <shortdesc>One-sentence summary for search and previews.</shortdesc>
  <conbody>
    <p>Definition and context.</p>
    <p>Key idea #1.</p>
    <p>Key idea #2.</p>
    <section>
      <title>Example</title>
      <p>Use an example to clarify.</p>
    </section>
  </conbody>
</concept>

Task template

<task id="...">
  <title>...</title>
  <shortdesc>What this procedure achieves.</shortdesc>
  <taskbody>
    <prereq>
      <p>Conditions and required permissions.</p>
    </prereq>
    <context>
      <p>When and why you do this.</p>
    </context>
    <steps>
      <step><cmd>Do step 1.</cmd></step>
      <step><cmd>Do step 2.</cmd></step>
      <step><cmd>Verify the result.</cmd></step>
    </steps>
    <result>
      <p>What success looks like.</p>
    </result>
  </taskbody>
</task>

Reference template

<reference id="...">
  <title>...</title>
  <shortdesc>What this reference covers.</shortdesc>
  <refbody>
    <section>
      <title>Syntax</title>
      <codeblock>command --option value</codeblock>
    </section>
    <section>
      <title>Options</title>
      <simpletable>
        <sthead>
          <stentry>Option</stentry>
          <stentry>Description</stentry>
        </sthead>
        <strow>
          <stentry><codeph>--help</codeph></stentry>
          <stentry>Displays help text.</stentry>
        </strow>
      </simpletable>
    </section>
  </refbody>
</reference>

Authoring for reuse: designing “reusable chunks”

Reuse is not free. It requires deliberate boundaries.

What should be reusable?

  • warnings, cautions, safety notes,
  • prerequisites and environment statements,
  • legal/compliance text,
  • UI labels and product naming,
  • standard procedures used across many tasks,
  • shared troubleshooting steps,
  • consistent definitions.

What should usually NOT be reused?

  • highly contextual narrative,
  • content that changes often,
  • content that’s meaningful only in one location,
  • deep cross-referenced paragraphs that assume surrounding text.

“Reusable chunk” rules

A reusable chunk should:

  • make sense when read alone,
  • avoid “above/below” references,
  • avoid map-specific context,
  • be stable over time.

If it depends on context, keep it local.

Metadata: the overlooked authoring power tool

Metadata improves:

  • search relevance,
  • portal navigation,
  • translation handling,
  • analytics and reporting,
  • content governance.

Useful metadata patterns

  • <shortdesc> for topic previews.
  • <prolog> for author, version, review status.
  • <keywords> for taxonomy and search.
  • product/platform attributes for filtering.

Governance tip

Create a “metadata minimum standard”:

  • every topic must have <shortdesc> and at least one keyword,
  • every task must include prerequisites if relevant,
  • every reference must include syntax and examples.

These standards reduce variability.

Images, UI screenshots, and media management

Image authoring best practices

  • Keep images in a dedicated folder: media/images/.
  • Use consistent naming: ui-settings-panel.png.
  • Use alt text and/or text equivalents where required.
  • Prefer vector (SVG) where your pipeline supports it and where clarity matters.

Screenshot governance

Screenshots are expensive to maintain. DITA doesn’t solve that automatically. To reduce screenshot maintenance:

  • minimize screenshot dependence,
  • use callouts sparingly,
  • document UI actions in text with <uicontrol> elements,
  • standardize screenshot production workflow.

Translation and localization implications

DITA’s structure can reduce translation cost, but only if you:

  • reuse content instead of duplicating it,
  • keep topics stable,
  • avoid rewriting shared snippets frequently,
  • keep IDs stable (for reuse references).

Practical localization tips

  • Don’t embed locale-specific text in shared snippets unless that’s intentional.
  • Use keys for product names and terms that differ by region.
  • Avoid hard-coded date formats and culturally specific examples in globally reused topics.

Common authoring mistakes (and how to fix them)

Mistake 1: “Generic topics everywhere”

Fix: adopt information typing (concept/task/reference) and enforce it via review.

Mistake 2: “Giant topics”

Fix: split into smaller topics and assemble with maps.

Mistake 3: “Conref everywhere”

Fix: reuse only stable, context-independent chunks; document ownership and change control.

Mistake 4: “Uncontrolled attributes”

Fix: define a controlled vocabulary and enforce it (subject scheme + process).

Mistake 5: “IDs keep changing”

Fix: treat IDs as stable contracts; manage renames with care.

A practical authoring workflow for teams

Here’s a workflow that works for both file-based and CCMS environments:

  1. Plan: create map skeleton (TOC) and topic list.
  2. Draft: create topic stubs with titles and shortdesc.
  3. Fill: author content following templates.
  4. Review: SME + editorial; check structure and semantics.
  5. Reuse pass: consolidate repeated content into reuse library.
  6. Metadata pass: keywords, attributes, taxonomy.
  7. Validate: run builds and resolve warnings/errors.
  8. Publish: produce HTML/PDF; confirm navigation and search behavior.
  9. Maintain: version and update with stable IDs and reuse targets.

Action checklist (do this to get productive quickly)

  • [ ] Choose topic types you will use (start with concept/task/reference).
  • [ ] Define naming conventions for files and IDs.
  • [ ] Create templates and enforce them (even if informally at first).
  • [ ] Establish reuse boundaries and a reuse folder.
  • [ ] Define attribute vocabulary for platform/audience/product.
  • [ ] Run validation on every change (locally or in CI).
  • [ ] Keep topics small and stable; assemble with maps.

Analytics and Instrumentation

Track authoring patterns and template usage with analytics. Monitor which templates are most used and where authors encounter issues. Use this data to improve authoring workflows and training.

Personalization and Conditional Processing

Use DITA conditional attributes to tailor authoring guidance for different roles or workflows. Example:

<section audience="editor">Editor-specific guidance here.</section>

Integrating with Conversational AI and NLP

Expose authoring best practices to chatbots or virtual assistants. Enable users to ask for authoring help and receive context-aware answers from this topic.

Continuous Improvement and Feedback Loops

Collect feedback from authors and editors on templates and workflows. Use analytics and feedback to refine authoring guidance and improve documentation quality.
