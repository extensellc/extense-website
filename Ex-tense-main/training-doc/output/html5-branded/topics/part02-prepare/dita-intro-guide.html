DITA at a Glance

Darwin Information Typing Architecture (DITA) is an XML-based standard for writing topic-oriented, reusable technical content and publishing it to multiple outputs (HTML, PDF, help systems, portals, etc.). It’s maintained as an open standard by the OASIS DITA Technical Committee.

Why DITA exists (the problems it solves)

DITA is designed for teams that need:

  • Reuse at scale (the same warning/procedure used across products and manuals)
  • Multiple deliverables from one source (HTML + PDF + embedded help)
  • Product variants (editions, platforms, regions, customer-specific builds)
  • Long-lived documentation with frequent updates
  • Large teams needing structure, consistency, and governance

The core idea: write small, semantically meaningful modules (topics), assemble them with maps, and publish through a repeatable pipeline.

The DITA mental model

Topics: small, typed units of information

DITA encourages authors to write content as topics: self-contained units that answer a user need. The most common base topic types are:

  • Concept — explain “what/why”
  • Task — explain “how”
  • Reference — provide “facts/specs/options”

This style is called information typing: using different structures depending on the user intent.

Maps: assemble topics into deliverables

A DITA map is how you organize and publish:

  • Defines the table of contents
  • Controls navigation structure
  • Can define relationships/links
  • Defines keys (late-bound references)
  • Often defines publishing context (for filtering/branding)

“Darwin”: specialization and evolution

DITA is built to evolve without breaking. You can extend (specialize) elements and constraints to fit your domain while keeping compatibility with tooling.

XML basics for DITA authors

DITA files are written in XML. If you have never worked with XML before, here are the five things you need to know before reading any DITA code:

  1. Elements are the building blocks. They have an opening tag and a closing tag: 
    <p>This is a paragraph.</p>
<title>My Topic Title</title>
    The text between the tags is the element's content. The tag name (like p or title) tells the processor what kind of content this is.
  2. Attributes add extra information to an element. They go inside the opening tag as name="value" pairs: 
    <concept id="my-topic" xml:lang="en-US">
    Here id and xml:lang are attributes of the concept element.
  3. Nesting means putting elements inside other elements. Indentation shows the hierarchy: 
    <ul>
  <li>First item</li>
  <li>Second item</li>
</ul>
    The li elements are nested inside the ul element. Not every element can go inside every other element — DITA's rules (defined by its DTD or schema) control what nesting is allowed.
  4. Self-closing tags are for elements with no text content. Instead of <tag></tag>, write <tag/>: 
    <data name="difficulty" value="beginner"/>
<indexterm>DITA</indexterm>  <!-- has content, so NOT self-closing -->
  5. The DOCTYPE declaration at the top of every DITA file tells the processor which topic type this file uses: 
    <!DOCTYPE concept PUBLIC "-//OASIS//DTD DITA Concept//EN" "concept.dtd">
    You do not need to memorize this — copy it from a template. The key part is the word after DOCTYPE (concept, task, reference) which must match your root element.

That is all the XML you need to start reading and writing DITA. Every code example in this guide uses these five patterns.

Key building blocks in the spec (what you’ll see in real projects)

Topic anatomy (common elements)

Most topics include:

  • <title>: topic title
  • Body element depends on type: <conbody>, <taskbody>, <refbody>
  • Common inline structures: <p>, <ph>, <uicontrol>, <codeph>, etc.
  • Lists/tables: <ul>, <ol>, <simpletable>, <table>

Task topics add specialized structure:

  • <prereq> (before you start)
  • <context> (why/when)
  • <steps> with <step><cmd>...</cmd></step>
  • <result> / <postreq>

Reference topics commonly use:

  • <refbody> with sections or a reference table-like structure
  • <properties> (property/value definitions) in some models

Maps: topicref and hierarchy

The map primarily uses <topicref> to point to topics:

<map>
  <title>Getting Started</title>
  <topicref href="concepts/what-is-dita.dita"/>
  <topicref href="tasks/install.dita"/>
</map>

DITA maps can also use:

  • <topichead>: a heading without a linked topic
  • <topicgroup>: grouping without appearing in TOC (tool-dependent behavior)
  • Relationship tables (advanced linking)

Domains, constraints, and specialization (practical view)

In real implementations you may see:

  • Domains: add sets of elements (e.g., UI domain, highlighting domain)
  • Constraints: restrict what’s allowed for a certain doc set
  • Specialization: define new elements as specializations of existing ones

Rule of thumb: specialize only when you have a strong semantic or validation need and you can support it in publishing + authoring tools.

The most important DITA features you’ll actually use

Reuse with conref (content reference)

conref pulls content from another location.

Use cases

  • Shared warnings, notes, prerequisites
  • Shared UI labels
  • Shared steps or step fragments

Example (reuse a note):

<p conref="shared/common-notes.dita#common-notes/install-note"/>

Best practice

  • Keep conref targets stable and well-named
  • Treat shared snippets as “mini APIs” (don’t casually restructure them)

Keys + keyref (late-bound reuse and linking)

Keys are defined in the map; topics reference them with keyref.

Why keys matter:

  • Change a URL/product name once in the map → updates everywhere
  • Reuse topics in different publications with different key definitions
  • Avoid hard-coding product names and links in topics

Example (in map):

<keydef keys="product-name">
  <topicmeta>
    <keywords><keyword>Acme Writer Pro</keyword></keywords>
  </topicmeta>
</keydef>

<keydef keys="support-url" href="https://example.com/support"/>

In a topic:

<p>For help with <ph keyref="product-name"/>, see <xref keyref="support-url"/>.</p>

Conditional processing (profiling / filtering)

DITA supports filtering content based on attributes. Common attributes include:

  • audience
  • platform
  • product
  • props (generic)
  • otherprops (legacy-ish patterns; depends on governance)

Typical use case: one source, multiple editions (e.g., Windows vs Linux).

In topics:

<p platform="windows">Click Start > Settings.</p>
<p platform="linux">Open the system settings panel.</p>

Then apply a filter (often using a DITAVAL file in DITA-OT) to include/exclude.

Governance tip: Define a controlled vocabulary early (what values are allowed) and enforce it.

A tiny working example (topics + map)

Concept topic (concepts/what-is-dita.dita)

<?xml version="1.0" encoding="UTF-8"?>
<!DOCTYPE concept PUBLIC "-//OASIS//DTD DITA Concept//EN" "concept.dtd">
<concept id="what-is-dita">
  <title>What is DITA?</title>
  <conbody>
    <p>DITA is an XML-based standard for writing topic-oriented content that can be reused and published to multiple formats.</p>
  </conbody>
</concept>

Task topic (tasks/install.dita)

<?xml version="1.0" encoding="UTF-8"?>
<!DOCTYPE task PUBLIC "-//OASIS//DTD DITA Task//EN" "task.dtd">
<task id="install">
  <title>Install the product</title>
  <taskbody>
    <prereq>
      <p>Make sure you have administrator access.</p>
    </prereq>
    <steps>
      <step><cmd>Download the installer.</cmd></step>
      <step><cmd>Run the installer and follow the prompts.</cmd></step>
      <step><cmd>Verify installation by launching the application.</cmd></step>
    </steps>
  </taskbody>
</task>

Map (guides/getting-started.ditamap)

<?xml version="1.0" encoding="UTF-8"?>
<!DOCTYPE map PUBLIC "-//OASIS//DTD DITA Map//EN" "map.dtd">
<map>
  <title>Getting Started</title>
  <topicref href="../concepts/what-is-dita.dita" type="concept"/>
  <topicref href="../tasks/install.dita" type="task"/>
</map>

Publishing with DITA Open Toolkit (DITA-OT) (common approach)

DITA-OT is the most widely used open-source engine for publishing DITA.

Basic commands (typical)

Generate HTML5:

dita -i guides/getting-started.ditamap -f html5 -o out/html

Generate PDF (toolchain dependent—usually FO-based):

dita -i guides/getting-started.ditamap -f pdf -o out/pdf

Repeatable builds with project files (recommended)

As doc sets grow, use project files to store:

  • inputs
  • outputs
  • filters
  • parameters

This makes builds consistent across machines/CI.

Where customization lives

Common customization layers include:

  • Parameters (simple switches)
  • CSS/HTML templates (HTML output)
  • PDF plugins/themes (PDF output)
  • Custom XSLT and DITA-OT plugins (advanced)

Practical advice: start with configuration + small overrides, then grow into plugins only when needed.

Authoring workflow: how teams actually work with DITA

Folder structure (simple and scalable)

A common approach:

dita/
  maps/
    guides/
      getting-started.ditamap
  topics/
    concepts/
      what-is-dita.dita
    tasks/
      install.dita
    reference/
  reuse/
    common-notes.dita
  media/
    images/
  ditaval/
    windows.ditaval
    linux.ditaval
  out/

Content development lifecycle (CDLC)

Many teams use a lifecycle like:

  1. Plan IA (maps, deliverables, variants)
  2. Author topics (semantics and structure)
  3. Reuse and link (conref, keys)
  4. Validate (XML validity + broken refs)
  5. Publish (local + CI)
  6. Review (SME + editorial)
  7. Release and maintain

CCMS vs file-based DITA

  • File-based: Git + DITA files, fast for engineers/DevOps-heavy teams
  • CCMS: workflow, reuse management, metadata, translations, governance at enterprise scale

DITA works in both models; the difference is management tooling.

Best practices (what prevents pain later)

Write small topics with a single purpose

  • Don’t write “chapters” as a single topic
  • Prefer “one user goal = one topic”

Use keys for anything that changes across deliverables

  • Product names, external URLs, reused references
  • This keeps topics clean and portable

Be deliberate about conref

  • Great for stable reusable fragments
  • Overuse can make content hard to read and maintain
  • Consider “reuse boundaries”: warnings, UI strings, legal text, standard steps

Standardize attribute vocabularies

  • Define allowed values for platform/audience/product
  • Enforce via review and/or constraints

Establish naming conventions

  • Topic IDs and file names should be stable
  • Use predictable IDs for conref targets

Validation and build hygiene

  • Always validate during CI:

- well-formed XML - schema/DTD validation (as applicable) - broken links/conrefs/keys

Common pitfalls (and how to avoid them)

  • Giant topics → split into focused topics; assemble via maps
  • Hard-coded product names/URLs → use keys (keyref)
  • Conref spaghetti → limit conref usage; centralize shared content; document ownership
  • Uncontrolled filtering values → standardize and govern values
  • Specialization too early → start with base DITA; specialize only with clear requirements

Quick-start checklist

  1. Decide your deliverables (HTML? PDF? both?)
  2. Define a starter map set (Getting Started, User Tasks, Reference)
  3. Author 5–10 topics using concept/task/reference
  4. Introduce keys for product name + URLs
  5. Add one reuse file (reuse/common-notes.dita) and use a couple of conrefs
  6. Set up one filtering example (platform=windows/linux) + build it both ways
  7. Automate publishing via a script or CI
