Publishing Automation Guide

CI/CD for documentation: containerized builds, multi-format output, validation gates, and AI-ready export — from a single source, on every commit.

Manual Publishing Is Broken

If any of these sound familiar, you have a publishing automation problem.

Before: The Manual Process

  • A writer runs DITA-OT locally on their laptop — different Java version, different plugins, different results
  • PDF generation takes 15–40 minutes; nobody runs it until "release day"
  • Broken links, missing images, and invalid XML are discovered after the build completes
  • Someone emails the PDF to the reviewer. Feedback comes back in a Word doc
  • The branded PDF template was last updated by a contractor who left 2 years ago
  • There is no build for chatbot-ready JSONL because "that's an AI team problem"

After: The Automated Pipeline

  • Every commit triggers a containerized build — same DITA-OT version, same plugins, same output everywhere
  • HTML5, branded PDF, and chatbot-ready JSONL are generated in parallel in under 3 minutes
  • Validation gates catch broken links, invalid XML, missing metadata, and terminology issues before merge
  • Reviewers get a staging URL. Feedback is a PR comment
  • Branding is a versioned DITA-OT plugin, maintained in Git like any other codebase artifact
  • AI-ready JSONL is a first-class output format, built and deployed alongside HTML and PDF

The Automated Pipeline

Eight stages from commit to production. Every stage runs in a Docker container for reproducibility.

1

Git Push

Writer commits to a feature branch. PR triggers the pipeline automatically.

2

Schema Validation

DITA DTD/RNG validation + Schematron business rules. Invalid XML blocks the build immediately.

3

Link & Reference Check

Every xref, href, conref, and keyref is verified. Broken references fail the build before it wastes time generating output.

4

Terminology & Style

Vale or Acrolinx checks language consistency against your style guide. Flags passive voice, banned terms, and inconsistent capitalization.

5

Build All Formats

DITA-OT generates HTML5, branded PDF, and JSONL in parallel. Custom plugins handle company branding, navigation, and metadata enrichment.

6

Image & Asset Audit

Missing alt text, oversized files (>500KB), orphaned graphics, and unlicensed stock images are flagged post-build.

7

Deploy to Staging

HTML5 output is deployed to a staging URL. Reviewers see the actual rendered output, not raw XML.

8

Promote to Production

After PR approval, output is deployed to S3/Netlify/CDN. PDF is uploaded to the doc portal. JSONL is pushed to the vector store.

Containerized Builds with Docker

The single most impactful change you can make to your publishing pipeline. No more "it works on my machine."

Why Docker?

DITA-OT depends on Java, Saxon, Apache FOP, and a stack of plugins — each with version-specific behavior. When Writer A runs DITA-OT 4.1 with Java 17 and Writer B runs DITA-OT 3.7 with Java 11, you get different output from the same source. In regulated industries (defense, life sciences, financial services), this is a compliance failure.

A Docker image locks the entire toolchain — DITA-OT version, Java version, custom plugins, fonts, and configuration — into a single, versioned artifact. The pipeline pulls that image and runs the build. Every build. Every environment. Identical output.

What's in the Image

  • Base: Eclipse Temurin JDK 17 (slim)
  • DITA-OT: Pinned version (e.g., 4.2.3) installed via the official distribution
  • Custom Plugins: Branded PDF plugin, HTML5 theme, JSONL exporter — all baked in
  • Fonts: Corporate fonts for PDF generation (no more "font not found" on the build server)
  • Validation Tools: Vale CLI, xmllint, custom Schematron rules
  • Build Script: Gradle wrapper with tasks for each output format
# Dockerfile — Production DITA-OT build image
FROM eclipse-temurin:17-jdk-jammy AS base

# Install DITA-OT
ARG DITA_OT_VERSION=4.2.3
RUN curl -sL https://github.com/dita-ot/dita-ot/releases/download/${DITA_OT_VERSION}/dita-ot-${DITA_OT_VERSION}.zip \
    -o /tmp/dita-ot.zip && unzip /tmp/dita-ot.zip -d /opt
ENV DITA_HOME=/opt/dita-ot-${DITA_OT_VERSION}
ENV PATH="${DITA_HOME}/bin:${PATH}"

# Install custom plugins
COPY plugins/ ${DITA_HOME}/plugins/
RUN dita install

# Install corporate fonts for PDF
COPY fonts/ /usr/share/fonts/custom/
RUN fc-cache -fv

WORKDIR /workspace
ENTRYPOINT ["dita"]

Pipeline Usage

Once the image is built and pushed to your container registry, every CI job uses the same image:

docker run --rm -v $(pwd):/workspace your-registry/dita-ot:4.2.3 -i main.ditamap -f html5 -o /workspace/out/html5

GitHub Actions, Azure DevOps, Jenkins, and GitLab CI all support container-based jobs natively. The CI platform pulls the image, mounts the source, and runs the build. No Java installation, no plugin installation, no configuration drift.

Multi-Format Output from a Single Source

One commit, four formats, one pipeline. Including chatbot-ready JSONL as a first-class output.

Branded HTML5

Responsive, searchable web output with your company's visual identity. Custom DITA-OT HTML5 plugin handles navigation, breadcrumbs, version selector, and search integration. Deployed to S3/Netlify/CDN with cache-busting asset fingerprints.

Typical build: 200 topics → 45 seconds

Branded PDF

Print-ready PDF using Apache FOP or Antenna House with your brand's fonts, colors, headers/footers, and cover page. Generated from a custom PDF plugin — not a post-processing step. Includes auto-generated TOC, index, and cross-reference page numbers.

Typical build: 200 topics → 90 seconds

Chatbot-Ready JSONL

Section-aware chunks with full metadata inheritance: topic type, audience, difficulty, domain, intent, section ID. Each chunk is a self-contained unit ready for vector store embedding. Includes a glossary index and content manifest for corpus validation.

See: AI Content Readiness Guide for the metadata schema.

Additional Formats

The same pipeline can produce EPUB (for offline readers), Markdown (for developer portals), normalized DITA (for CCMS ingestion), and custom XML transforms (for data interchange). Each format is a Gradle task — adding a new format is a one-line configuration change.

Pipeline total: All 4 formats in parallel → under 3 minutes

// build.gradle — parallel multi-format build
task buildAll {
  dependsOn 'buildHtml5', 'buildPdf', 'buildJsonl'
}

task buildHtml5(type: Exec) {
  commandLine 'dita', '-i', 'main.ditamap', '-f', 'com.extense.html5',
    '-o', 'out/html5', '-Dargs.css=brand.css'
}

task buildPdf(type: Exec) {
  commandLine 'dita', '-i', 'main.ditamap', '-f', 'com.extense.pdf',
    '-o', 'out/pdf'
}

task buildJsonl(type: Exec) {
  commandLine 'dita', '-i', 'main.ditamap', '-f', 'com.extense.jsonl',
    '-o', 'out/jsonl', '-Dchunk.strategy=section-aware'
}

Supported CI/CD Platforms

We configure pipelines on the platform you already use. Here's what each one looks like.

GitHub Actions (Recommended Starting Point)

# .github/workflows/docs.yml
name: Build & Deploy Docs
on:
  push: { branches: [main] }
  pull_request: { branches: [main] }

jobs:
  build:
    runs-on: ubuntu-latest
    container: your-registry/dita-ot:4.2.3
    steps:
      - uses: actions/checkout@v4

      - name: Validate XML
        run: xmllint --noout --dtdvalid $DITA_HOME/dtd/topic.dtd src/**/*.dita

      - name: Check links
        run: dita -i main.ditamap -f lint --check.links=true

      - name: Build HTML5
        run: dita -i main.ditamap -f com.extense.html5 -o out/html5

      - name: Build PDF
        run: dita -i main.ditamap -f com.extense.pdf -o out/pdf

      - name: Build JSONL
        run: dita -i main.ditamap -f com.extense.jsonl -o out/jsonl

      - name: Deploy to S3
        if: github.ref == 'refs/heads/main'
        run: aws s3 sync out/html5 s3://docs.example.com/ --delete

Azure DevOps

Multi-stage YAML pipelines with artifact management and approval gates. Container jobs pull the same Docker image. Ideal for enterprises already on Azure — integrates with Azure Blob Storage for hosting and Azure AI Search for vector retrieval.

Jenkins

Declarative or scripted Groovy pipelines running on self-hosted agents. Maximum control over infrastructure. Our pick for air-gapped defense environments where SaaS CI platforms are prohibited. Supports Docker agent blocks for containerized builds.

GitLab CI

Integrated CI/CD with built-in container registry and merge-request pipelines. Great for teams already using GitLab for source control — the pipeline, container registry, and Pages hosting are all in one platform.

Validation Gates

A publish pipeline that doesn't validate is just a faster way to ship broken documentation. We configure quality gates that block bad builds.

Schema & Schematron

DTD/RNG validation catches structural errors (missing required elements, wrong nesting). Schematron rules enforce business-level constraints: naming conventions, required metadata fields, prohibited elements, maximum topic length, and image resolution requirements.

Link & Reference Integrity

Every xref, href, conref, and keyref is resolved and verified at build time. Missing targets, circular references, and dead external URLs fail the build. This catches the #1 source of broken documentation: refactoring topics without updating cross-references.

Terminology & Style

Integration with Vale (open source) or Acrolinx (enterprise) for automated style checking. Custom rules enforce your style guide: banned terms, passive voice limits, abbreviation consistency, and product name capitalization. Runs as a pre-build check.

Image & Asset Audit

Missing alt text, oversized files (>500KB), orphaned graphics (referenced but not committed), and unlicensed stock images are flagged. Ensures accessibility compliance and prevents repo bloat from uncompressed screenshots.

Metadata Completeness

Every topic must have all required metadata fields populated (audience, difficulty, domain, intent) with values from the controlled vocabulary. Topics with missing or invalid metadata are rejected — because incomplete metadata breaks AI retrieval downstream.

Build Notifications

Slack, Teams, or email alerts on build failure with a direct link to the failing step and the specific error. Build dashboards show historical pass/fail rates, average build time, and output size trends. No more "did anyone check if the build passed?"

Custom DITA-OT Plugins

Off-the-shelf DITA-OT output is a starting point. Production documentation needs custom branding, navigation, and export formats.

Branded PDF Plugin

Custom XSL-FO transforms for Apache FOP or Antenna House. Handles cover pages, headers/footers with logos, corporate fonts, color schemes, auto-generated TOC and index, and conditional content filtering by audience or product variant.

Custom HTML5 Theme

Responsive web output with your company's design system. Navigation sidebar, breadcrumbs, version selector, search integration (Algolia, Elasticsearch, or Lunr.js), and analytics tracking. Deployed as a static site — no server infrastructure required.

JSONL Exporter

Section-aware chunking with full metadata inheritance for vector store ingestion. Produces chunks.jsonl, glossary-index.json, and content-manifest.json. Configurable chunking strategy: by section, by topic, or by semantic boundary.

Plugin Development & Maintenance

Every custom plugin is a versioned, testable artifact in your Git repository — not a one-off customization buried in someone's local DITA-OT installation. We build plugins with automated tests, document the configuration parameters, and train your team to maintain them. When DITA-OT releases a new version, we validate plugin compatibility and update as needed.
